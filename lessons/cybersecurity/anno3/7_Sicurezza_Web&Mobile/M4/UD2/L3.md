# L3 — Vulnerabilità dei Cookie e Contromisure

> 📌 I cookie rendono HTTP stateful, ma ereditano molte debolezze del contesto in cui viaggiano: se HTTP è in chiaro possono essere intercettati o modificati, se sono accessibili lato client possono essere rubati via script, e se contengono dati applicativi modificabili possono essere manipolati dall'utente.

---

### **1. Obiettivo della Lezione**

In questa lezione vengono evidenziate le principali vulnerabilità del meccanismo dei cookie.

Nella lezione precedente abbiamo visto che i cookie servono a compensare il fatto che HTTP sia stateless.

In questa lezione vediamo il lato opposto:
- i cookie possono essere letti se viaggiano in chiaro;
- i cookie possono essere modificati o cancellati;
- i cookie possono essere manipolati direttamente lato client;
- i cookie possono essere rubati tramite script;
- le alternative ingenue, come i campi nascosti nei form, soffrono di problemi simili;
- esistono contromisure come `Secure` e `HttpOnly`, ma non risolvono ogni problema.

> 💡 Un cookie di sessione va trattato come una credenziale temporanea: se l'attaccante lo ottiene o lo modifica, può influenzare lo stato applicativo dell'utente.

---

### **2. Intercettazione dei Cookie su HTTP**

I cookie si basano sul protocollo HTTP.

Se HTTP viene usato in chiaro, anche gli header HTTP vengono trasmessi in chiaro.

Questo include:
- header `Cookie` nelle richieste;
- header `Set-Cookie` nelle risposte;
- eventuali valori di sessione;
- identificatori univoci;
- preferenze;
- altri dati applicativi.

Un attaccante che intercetta il traffico può leggere i cookie trasmessi.

#### **2.1. Perché è Pericoloso**

Se il cookie identifica una sessione autenticata, l'attaccante può usarlo per impersonare l'utente.

Per esempio:

```http
Cookie: sessionId=abc123
```

Se Trudy legge `sessionId=abc123`, può inviare richieste al server includendo lo stesso cookie.

Il server potrebbe associarla alla sessione della vittima.

Questo tipo di problema è alla base di attacchi di furto di sessione.

> ⚠️ Se il cookie è sufficiente a identificare una sessione, rubare il cookie può equivalere a rubare l'identità applicativa dell'utente.

---

### **3. Contromisura: HTTPS e Flag Secure**

La contromisura principale contro l'intercettazione è usare HTTPS.

HTTPS protegge la comunicazione tra browser e server cifrando il traffico.

Per i cookie, è importante anche impostare il flag:

```http
Secure
```

Un cookie `Secure` viene inviato dal browser solo su connessioni HTTPS.

Esempio:

```http
Set-Cookie: sessionId=abc123; Secure
```

#### **3.1. Perché Serve `Secure` anche se il Sito Usa HTTPS**

Non basta che il sito supporti HTTPS.

Se il cookie non ha il flag `Secure`, il browser potrebbe inviarlo anche in una richiesta HTTP verso lo stesso dominio, se tale richiesta viene effettuata.

Questo esporrebbe il cookie in chiaro.

Il flag `Secure` impedisce questo comportamento.

#### **3.2. Limiti Storici e Implementativi**

Il transcript sottolinea che storicamente l'uso di HTTPS non è sempre stato una garanzia assoluta.

Per ragioni di efficienza, alcune implementazioni o configurazioni potevano:
- non proteggere tutto il traffico;
- usare HTTPS solo per alcuni messaggi;
- proteggere solo la fase iniziale;
- lasciare esposti alcuni dati.

Nelle implementazioni moderne, se `Secure` è impostato correttamente e HTTPS è configurato correttamente, il cookie viene trasmesso dentro il canale cifrato.

> 📌 `Secure` lega il cookie al canale HTTPS: è una protezione essenziale per cookie di sessione e cookie sensibili.

---

### **4. Cookie Poisoning**

Un'altra vulnerabilità è il **cookie poisoning**, cioè l'avvelenamento del cookie.

Consiste nella modifica, cancellazione o alterazione dei valori contenuti nei cookie.

L'attaccante può tentare di cambiare:
- identificatori;
- ruoli;
- preferenze;
- prezzi;
- quantità;
- autorizzazioni;
- flag applicativi;
- dati di sessione.

#### **4.1. Modifica del Cookie nel Traffico**

Se il traffico HTTP è in chiaro, un attaccante attivo può intercettare una richiesta e modificare l'header:

```http
Cookie: nome=valore
```

prima che arrivi al server.

Il problema deriva dal fatto che, senza protezione crittografica, request e response possono essere:
- lette;
- modificate;
- cancellate;
- reinviate.

#### **4.2. Modifica del File dei Cookie**

Il cookie è memorizzato lato client.

Questo significa che l'attaccante non deve essere necessariamente un soggetto esterno sulla rete.

Può essere anche l'utente stesso.

Un utente malevolo può provare a modificare direttamente i cookie salvati dal browser, per ottenere un vantaggio.

> ⚠️ Non bisogna assumere che il client sia fidato. L'utente può leggere, cancellare o modificare dati memorizzati sul proprio dispositivo.

---

### **5. Esempio: Carrello della Spesa Implementato nei Cookie**

Consideriamo un vecchio modello di applicazione e-commerce in cui il carrello viene memorizzato interamente nel cookie.

Supponiamo che il cookie contenga il totale da pagare:

```http
Cookie: totale=150
```

Un utente malevolo potrebbe modificare il file dei cookie e sostituire il valore:

```http
Cookie: totale=15
```

Se il server si fidasse del valore ricevuto dal client, l'utente potrebbe pagare 15 invece di 150.

#### **5.1. Perché l'Esempio non è più il Modello Corrente**

Il transcript sottolinea che questo esempio non rappresenta il modello attuale delle grandi applicazioni.

Oggi, di norma, non si memorizzano nel cookie tutte le informazioni del carrello.

Il cookie contiene piuttosto un identificatore:

```http
Cookie: cartId=xyz789
```

Il carrello vero è memorizzato lato server.

Il server usa `cartId` per recuperare il carrello dal database di backend.

#### **5.2. Regola di Progettazione**

Il server non deve fidarsi di valori critici provenienti dal client.

Valori come:
- prezzo;
- totale;
- ruolo utente;
- permessi;
- saldo;
- sconto;
- quantità finale;

devono essere verificati o mantenuti lato server.

> 📌 I cookie possono trasportare identificatori, ma non dovrebbero essere l'unica fonte di verità per dati critici o modificabili dall'utente.

---

### **6. Accesso ai Cookie tramite Script**

Un'altra vulnerabilità deriva dal fatto che i cookie possono essere accessibili da linguaggi di scripting lato client.

In JavaScript, se il cookie non è protetto da `HttpOnly`, può essere letto con:

```javascript
document.cookie
```

Questo significa che uno script eseguito nel contesto della pagina può leggere i cookie relativi al dominio.

Il problema diventa grave se l'attaccante riesce a far eseguire uno script malevolo nel browser della vittima.

Questo è il collegamento con gli attacchi di tipo **Cross-Site Scripting**.

---

### **7. Esempio di Furto Cookie tramite Script**

Supponiamo che esista un sito:

```text
dictionary.com
```

che permette all'utente di cercare una parola tramite un form.

Il browser invia una richiesta con metodo GET:

```http
GET /search.php?term=parola HTTP/1.1
Host: dictionary.com
```

Il server esegue lo script `search.php`.

Lo script genera una pagina HTML che contiene:
- il risultato della ricerca;
- il termine cercato dall'utente.

#### **7.1. Implementazione Vulnerabile**

Il problema nasce se il server reinserisce nella pagina HTML il valore ricevuto dall'utente senza sanitizzarlo.

Per esempio, se il parametro `term` viene copiato direttamente nella risposta HTML.

L'applicazione si aspetta una parola normale:

```text
term=casa
```

ma l'attaccante può costruire un link in cui `term` contiene uno script.

#### **7.2. Link Malevolo**

Un utente malevolo può costruire un URL che include, nella query string, un tag `<script>`.

L'utente viene convinto a cliccare il link, per esempio tramite phishing.

Quando la vittima clicca:
- il browser invia la richiesta a `dictionary.com`;
- il server genera una risposta HTML contenente il valore di `term`;
- il valore di `term` contiene uno script;
- il browser interpreta la pagina;
- durante il parsing, esegue lo script.

#### **7.3. Invio dei Cookie all'Attaccante**

Lo script malevolo può usare:

```javascript
document.cookie
```

per leggere i cookie della vittima relativi a `dictionary.com`.

Poi può inviare quei cookie a un dominio controllato dall'attaccante, per esempio:

```text
badguy.com
```

In questo modo l'attaccante ottiene i cookie dell'utente.

#### **7.4. Perché l'Attacco Funziona**

L'attacco funziona perché:
- il server inserisce input utente nella pagina senza sanitizzazione;
- il browser interpreta lo script come parte della pagina legittima;
- lo script viene eseguito nel contesto del dominio vulnerabile;
- `document.cookie` permette di leggere i cookie accessibili a JavaScript;
- il cookie può poi essere inviato all'attaccante.

> ⚠️ Il browser non distingue automaticamente tra script scritto dallo sviluppatore e script iniettato tramite input utente se entrambi finiscono nella pagina HTML.

---

### **8. Contromisura: Flag HttpOnly**

Una contromisura specifica contro il furto via `document.cookie` è il flag:

```http
HttpOnly
```

Esempio:

```http
Set-Cookie: sessionId=abc123; HttpOnly
```

Se `HttpOnly` è impostato, il cookie:
- viene inviato nelle richieste HTTP o HTTPS;
- resta disponibile al server;
- non è accessibile tramite script lato client.

Quindi uno script malevolo non può leggerlo con:

```javascript
document.cookie
```

#### **8.1. HttpOnly non Elimina XSS**

`HttpOnly` non elimina la vulnerabilità XSS.

Riduce però uno degli effetti più gravi: il furto diretto del cookie di sessione.

Uno script malevolo potrebbe ancora:
- modificare la pagina;
- fare richieste a nome dell'utente;
- leggere dati presenti nella pagina;
- interagire con l'applicazione nel contesto della vittima.

Quindi `HttpOnly` è una difesa importante, ma non sostituisce la sanitizzazione dell'input e l'escaping dell'output.

> 📌 `HttpOnly` protegge il cookie dall'accesso JavaScript, ma l'applicazione deve comunque prevenire l'iniezione di script.

---

### **9. Tracking Cookie e Configurazione del Browser**

I cookie sono un meccanismo discusso anche per ragioni di privacy.

Nella lezione precedente abbiamo visto il problema dei tracking cookie.

Il problema persiste se l'utente non configura correttamente le preferenze del browser.

In particolare, l'utente può:
- bloccare cookie di terze parti;
- cancellare cookie periodicamente;
- usare impostazioni di privacy più restrittive;
- controllare quali siti possono impostare cookie.

Se queste impostazioni non vengono usate, i cookie di terze parti possono continuare a tracciare l'utente.

> ⚠️ Le contromisure tecniche esistono, ma se dipendono dalla configurazione dell'utente non sempre vengono applicate correttamente.

---

### **10. Alternative ai Cookie: Campi Nascosti nei Form**

Dato che i cookie hanno problemi di sicurezza e privacy, è naturale chiedersi se esistano alternative.

Una tecnica usata soprattutto nei primi anni 2000 era l'uso di campi nascosti nei form.

HTML permette di definire un input nascosto usando:

```html
<input type="hidden" name="cartTotal" value="150">
```

Il campo non viene visualizzato nella pagina.

Tuttavia viene incluso nei dati inviati al server quando il form viene sottomesso.

#### **10.1. Perché Sembrava Utile**

I campi hidden permettevano di mantenere stato tra più richieste senza usare cookie.

Per esempio:
- identificatore di sessione;
- valore del carrello;
- dati intermedi;
- informazioni applicative.

Il server inviava questi valori dentro la pagina HTML.

Il browser li rispediva al server alla sottomissione del form.

#### **10.2. Perché Sono Vulnerabili**

Il fatto che un campo sia nascosto nella pagina non significa che sia protetto.

L'utente può:
- visualizzare il sorgente HTML;
- modificare il DOM con strumenti di sviluppo;
- salvare e modificare la pagina;
- intercettare la request;
- cambiare il valore prima dell'invio.

Se viene usato il metodo GET, il valore può essere visibile anche nell'URL della richiesta.

Quindi i campi nascosti soffrono di problemi simili ai cookie.

> ⚠️ "Nascosto" nell'interfaccia non significa "sicuro". Tutto ciò che viene inviato al client può essere letto o modificato dal client.

---

### **11. Esempio: Carrello con Campo Hidden**

Un'applicazione vulnerabile potrebbe usare:

```html
<input type="hidden" name="total" value="150">
```

Quando l'utente conferma l'acquisto, il browser invia:

```http
POST /checkout HTTP/1.1
Host: shop.example

total=150
```

Un utente malevolo può modificare il valore:

```html
<input type="hidden" name="total" value="15">
```

e inviare:

```http
POST /checkout HTTP/1.1
Host: shop.example

total=15
```

Se il server accetta il valore senza ricalcolarlo lato server, l'attacco riesce.

Questo è lo stesso errore logico del carrello implementato interamente nei cookie.

#### **11.1. Attacchi Storici**

Il transcript sottolinea che, nonostante la vulnerabilità sia evidente, negli anni 2000 molte piattaforme di commercio elettronico usavano campi hidden per memorizzare informazioni critiche.

Molte hanno subito attacchi in cui l'utente modificava il valore del carrello o del prezzo.

> 📌 La regola è sempre la stessa: prezzo, totale e autorizzazioni devono essere calcolati o verificati lato server, non accettati dal client.

### **12. Confronto tra Cookie e Campi Hidden**

Cookie e campi hidden sono diversi tecnicamente, ma condividono un limite fondamentale.

Entrambi possono trasportare stato lato client.

Entrambi possono essere:
- letti;
- modificati;
- cancellati;
- manipolati prima dell'invio;
- usati per attacchi se il server si fida del loro contenuto.

La differenza è che:
- il cookie viene gestito dal browser e inviato automaticamente in base a dominio e path;
- il campo hidden è parte della pagina HTML e viene inviato con il form.

Dal punto di vista della sicurezza, nessuno dei due deve contenere valori critici non verificati.

> ⚠️ Se il server si fida ciecamente di dati conservati lato client, il problema non è il meccanismo scelto: è il modello di fiducia.

---

### **13. Contromisure Generali**

#### **13.1. Usare HTTPS**

HTTPS protegge la comunicazione da intercettazione e modifica sul canale.

Per cookie sensibili, deve essere combinato con `Secure`.

#### **13.2. Impostare `Secure`**

Il flag `Secure` impedisce l'invio del cookie su HTTP in chiaro.

È fondamentale per cookie di sessione.

#### **13.3. Impostare `HttpOnly`**

Il flag `HttpOnly` impedisce l'accesso al cookie tramite JavaScript.

È utile per ridurre il rischio di furto del cookie in presenza di XSS.

#### **13.4. Non Memorizzare Dati Critici nel Cookie**

Il cookie dovrebbe contenere identificatori casuali e non prevedibili, non dati critici modificabili dall'utente.

Valori come prezzo, saldo, ruolo e autorizzazioni devono essere gestiti lato server.

#### **13.5. Verificare sempre Lato Server**

Ogni dato ricevuto dal client deve essere considerato non fidato.

Il server deve verificare:
- formato;
- autorizzazioni;
- coerenza;
- firma o MAC se il dato deve essere protetto;
- validità temporale;
- appartenenza alla sessione corretta.

#### **13.6. Sanitizzare Input ed Eseguire Escape dell'Output**

Per prevenire il furto di cookie via script, bisogna impedire l'iniezione di codice JavaScript.

Questo richiede:
- validazione dell'input;
- escaping dell'output;
- separazione tra dati e codice;
- policy lato browser quando disponibili.

> ✅ `Secure` e `HttpOnly` sono difese importanti, ma non bastano se l'applicazione continua a fidarsi di dati client-side o a riflettere input non sanitizzato.

---

### **14. Riepilogo**

> ✅ I cookie sono vulnerabili perché viaggiano su HTTP e sono memorizzati lato client. Se il traffico è in chiaro, possono essere letti o modificati; se l'utente è malevolo, può manipolare direttamente i propri cookie; se l'applicazione riflette input non sanitizzato, uno script iniettato può leggere `document.cookie` e inviarlo a un attaccante. Le principali contromisure sono usare HTTPS, impostare `Secure`, impostare `HttpOnly`, non memorizzare dati critici nei cookie, verificare tutto lato server e prevenire XSS con validazione ed escaping. Alternative come i campi hidden nei form non risolvono il problema: anche se non visibili nella pagina, restano dati lato client e possono essere letti o modificati.
