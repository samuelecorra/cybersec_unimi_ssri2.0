# L1 — Protocollo HTTP: Funzionamento, Messaggi e Vulnerabilità

> 📌 HTTP è il protocollo applicativo usato per la comunicazione tra browser e web server. È semplice, testuale e basato su richiesta-risposta, ma nella sua forma base non garantisce confidenzialità, integrità forte né autenticazione robusta: per questo nelle comunicazioni reali si usa HTTPS.

---

### **1. Obiettivo della Lezione**

In questa lezione viene descritto il funzionamento del protocollo **HTTP**, cioè **HyperText Transfer Protocol**.

HTTP è il protocollo usato per la comunicazione tra:
- il browser sul computer client;
- il web server che fornisce pagine e risorse web.

La lezione affronta:
- posizione di HTTP nello stack di rete;
- modello client-server;
- oggetti web e pagine HTML;
- connessioni TCP persistenti e non persistenti;
- formato delle HTTP request;
- formato delle HTTP response;
- metodi HTTP principali;
- header di richiesta e risposta;
- codici di stato;
- debolezze di sicurezza di HTTP;
- passaggio a HTTPS;
- differenza storica rispetto a S-HTTP.

> 💡 HTTP è il protocollo che rende possibile la navigazione web, ma non nasce come protocollo sicuro: la sicurezza deve essere aggiunta tramite meccanismi ulteriori.

---

### **2. HTTP come Protocollo Applicativo**

HTTP è un protocollo di livello applicativo rispetto al modello ISO/OSI.

Definisce le regole che governano il trasferimento di pagine web e oggetti web dal server al client.

Più precisamente, HTTP definisce:
- il formato delle richieste inviate dal browser;
- il formato delle risposte inviate dal web server;
- i metodi disponibili;
- i codici di stato;
- gli header che descrivono richiesta e risposta;
- il modo in cui client e server scambiano oggetti web.

HTTP non si occupa direttamente di:
- instradamento IP;
- affidabilità della trasmissione;
- controllo della congestione;
- trasmissione fisica dei bit.

Per questi aspetti si appoggia ai livelli sottostanti dello stack TCP/IP.

#### **2.1. HTTP e TCP**

HTTP usa TCP come protocollo di trasporto.

In generale:
- il client apre una connessione TCP verso il server;
- la connessione viene stabilita verso una porta del server;
- il client e il server si scambiano messaggi HTTP lungo la connessione;
- al termine, la connessione può essere chiusa o mantenuta aperta.

Per HTTP non cifrato, la porta standard è:

$$80$$

Per HTTPS, come vedremo più avanti, la porta standard è:

$$443$$

> 📌 HTTP definisce il formato dei messaggi applicativi; TCP fornisce il canale affidabile su cui quei messaggi viaggiano.

---

### **3. Caratteristiche Generali di HTTP**

HTTP ha alcune caratteristiche fondamentali.

#### **3.1. Protocollo Client-Server**

HTTP segue il modello client-server.

Il **client** è tipicamente il browser.

Il suo ruolo è:
- richiedere oggetti web;
- ricevere risposte;
- interpretare HTML;
- visualizzare contenuti;
- richiedere ulteriori oggetti collegati alla pagina.

Il **server** è il web server.

Il suo ruolo è:
- restare in ascolto;
- ricevere richieste;
- recuperare o generare oggetti;
- inviare risposte al client.

#### **3.2. Protocollo Orientato agli Oggetti**

HTTP è orientato agli oggetti web.

Qui il termine "oggetto" non va inteso nel senso della programmazione a oggetti.

Un oggetto web può essere:
- un file HTML;
- un'immagine JPEG, PNG o altro formato;
- un file audio;
- uno script;
- un foglio di stile;
- un'applet o componente eseguibile;
- un documento;
- un contenuto generato dinamicamente.

Una pagina web è spesso composta da un file HTML di base più molti oggetti collegati.

#### **3.3. Protocollo Stateless**

HTTP è un protocollo **stateless**.

Questo significa che il server, nel modello base del protocollo, non mantiene automaticamente memoria delle richieste precedenti dello stesso client.

Ogni richiesta viene trattata come indipendente.

Se lo stesso client si collega più volte allo stesso server, ciascuna richiesta appare come una richiesta nuova, salvo meccanismi aggiuntivi come cookie, sessioni o autenticazione applicativa.

> ⚠️ Lo statelessness semplifica il protocollo, ma rende necessaria una gestione esplicita dello stato applicativo, per esempio tramite cookie e sessioni. Se questa gestione è debole, nascono molte vulnerabilità web.

---

### **4. Oggetti Web e Ruolo del Browser**

L'oggetto principale richiesto da un browser è spesso una pagina HTML.

Questa pagina HTML può includere riferimenti ad altri oggetti tramite URL.

Esempi di oggetti inclusi:
- immagini;
- script;
- fogli di stile;
- file audio;
- video;
- applet;
- altre pagine HTML;
- risorse generate da servizi diversi.

Il browser:
- riceve il file HTML;
- lo interpreta;
- legge i tag;
- formatta il contenuto;
- individua gli oggetti collegati;
- invia nuove richieste HTTP per recuperarli;
- visualizza il risultato finale all'utente.

Quindi caricare una pagina web non significa necessariamente inviare una sola richiesta HTTP.

Spesso significa inviare molte richieste:
- una per il file HTML principale;
- una per ciascuna immagine;
- una per ciascun foglio CSS;
- una per ciascuno script;
- una per altri oggetti collegati.

---

### **5. Apertura di una Sessione HTTP**

Una sessione HTTP, nel caso classico, inizia quando il client apre una connessione TCP verso il web server.

Il flusso generale è:
- il browser crea un socket verso il server;
- il server è in ascolto sulla porta 80;
- il server accetta la connessione TCP;
- client e server si scambiano messaggi HTTP;
- la connessione TCP viene chiusa o mantenuta aperta.

In forma concettuale:

$$Client \rightarrow Server : \text{apertura connessione TCP}$$

$$Client \rightarrow Server : HTTP\ request$$

$$Server \rightarrow Client : HTTP\ response$$

$$Client/Server : \text{chiusura o riuso della connessione}$$

Il dettaglio cambia a seconda del tipo di connessione:
- non persistente;
- persistente.

---

### **6. Connessioni Non Persistenti e Persistenti**

Il comportamento delle connessioni HTTP dipende anche dalla versione del protocollo.

#### **6.1. HTTP/1.0 e Connessioni Non Persistenti**

HTTP/1.0 gestiva solo connessioni non persistenti.

In una connessione non persistente:
- ogni richiesta richiede una nuova connessione TCP;
- ogni risposta usa quella connessione;
- dopo la risposta, la connessione viene chiusa.

Se una pagina HTML contiene dieci immagini, il browser deve aprire più connessioni:
- una per la pagina HTML;
- una per ciascuna immagine.

Questo introduce overhead.

Ogni connessione TCP richiede:
- apertura della connessione;
- gestione dello stato;
- allocazione di risorse lato client;
- allocazione di risorse lato server;
- chiusura della connessione.

#### **6.2. HTTP/1.1 e Connessioni Persistenti**

HTTP/1.1 introduce la possibilità di usare connessioni persistenti.

In una connessione persistente:
- la connessione TCP resta aperta;
- il client può inviare più richieste sulla stessa connessione;
- il server può inviare più risposte sulla stessa connessione;
- la connessione viene chiusa solo alla fine o dopo timeout.

Questo riduce:
- numero di connessioni TCP;
- overhead;
- carico lato server;
- carico lato client;
- latenza dovuta ad aperture ripetute.

> 📌 Le connessioni persistenti migliorano l'efficienza perché permettono di scaricare più oggetti web usando lo stesso socket TCP.

### **7. Esempio con Connessione Non Persistente**

Supponiamo che l'utente voglia visualizzare:

$$http://www.sito.com/index.html$$

La pagina `index.html` contiene:
- testo;
- riferimenti a 10 immagini.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **7.1. Richiesta della Pagina HTML**

L'utente inserisce l'URL nella barra del browser e preme invio.

Il browser:
- apre una connessione TCP verso `www.sito.com`;
- usa la porta 80;
- invia una richiesta HTTP per `index.html`.

Il server:
- accetta la connessione TCP;
- riceve la request;
- invia una response contenente `index.html`;
- chiude la connessione TCP, perché siamo nel caso non persistente.

#### **7.2. Richiesta delle Immagini**

Il browser riceve il file HTML e inizia il parsing.

Durante il parsing trova i riferimenti alle 10 immagini.

Per ciascuna immagine:
- apre una nuova connessione TCP;
- invia una nuova HTTP request;
- riceve una nuova HTTP response;
- chiude la connessione.

Quindi, per una pagina con 10 immagini, si aprono 11 connessioni:
- 1 per il file HTML;
- 10 per le immagini.

#### **7.3. Caso Persistente**

Con una connessione persistente, il server non chiude la connessione dopo la prima response.

Il client può quindi usare lo stesso socket TCP per richiedere:
- `index.html`;
- immagine 1;
- immagine 2;
- fino all'immagine 10.

Il risultato è un uso più efficiente della connessione.

> ⚠️ Il modo in cui il protocollo gestisce le connessioni ha anche impatto di sicurezza e disponibilità: molte connessioni aperte possono aumentare il carico sul server.

### **8. Formato della HTTP Request**

Una richiesta HTTP è un messaggio testuale.

È composta da:
- request line;
- header opzionali o obbligatori in alcuni casi;
- riga vuota;
- corpo opzionale.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **8.1. Request Line**

La prima riga è la **request line**.

È sempre presente.

Contiene tre informazioni:
- metodo richiesto;
- risorsa richiesta;
- versione del protocollo.

Forma generale:

```http
METHOD URL HTTP/versione
```

Esempio:

```http
GET /index.html HTTP/1.1
```

#### **8.2. Header della Request**

Dopo la request line possono comparire righe di intestazione.

Ogni header ha la forma:

```http
Nome-Campo: valore
```

Ogni riga termina con i caratteri di ritorno a capo previsti dal protocollo, cioè **CRLF**.

Gli header specificano informazioni aggiuntive, come:
- host richiesto;
- browser usato;
- tipi di contenuto accettati;
- lingua preferita;
- gestione della connessione;
- credenziali o autorizzazioni;
- condizioni legate alla cache.

#### **8.3. Riga Vuota e Corpo**

La fine degli header è indicata da una riga vuota.

Dopo la riga vuota può esserci:
- il corpo della richiesta;
- nessun corpo.

Molte richieste HTTP, soprattutto quelle con metodo GET, non hanno corpo.

Altre richieste, per esempio con metodo POST, usano il corpo per inviare dati.

> 📌 HTTP è testuale: request e response sono costruite come righe leggibili, separate da CRLF e organizzate in intestazioni e corpo.

---

### **9. Esempio di HTTP Request**

Un esempio di richiesta HTTP/1.1 può essere:

```http
GET /index.html HTTP/1.1
Host: www.sito.com
User-Agent: Mozilla/5.0
Accept: text/html,image/*,*/*
Accept-Language: it
Accept-Charset: utf-8
Connection: keep-alive
```

#### **9.1. Campo Host**

In HTTP/1.1 il campo `Host` è obbligatorio.

Se manca, il server deve restituire un errore.

Il campo `Host` completa l'URL parziale presente nella request line.

Per esempio:

```http
GET /index.html HTTP/1.1
Host: www.sito.com
```

identifica la risorsa:

$$http://www.sito.com/index.html$$

Questo è importante perché più siti possono essere ospitati sullo stesso server fisico.

#### **9.2. Campo User-Agent**

Il campo `User-Agent` specifica il browser o il client usato.

Può servire per:
- statistiche;
- debugging;
- log lato server;
- adattamento del contenuto;
- analisi di compatibilità.

#### **9.3. Campi Accept**

I campi che contengono `Accept` servono alla negoziazione tra client e server.

Permettono al browser di indicare:
- tipi di oggetto accettabili;
- lingua preferita;
- set di caratteri;
- formati supportati.

Esempi:
- `Accept`;
- `Accept-Language`;
- `Accept-Charset`;
- `Accept-Encoding`.

#### **9.4. Campo Connection**

Il campo `Connection` indica come gestire la connessione TCP.

Valori tipici:
- `close`;
- `keep-alive`.

`close` indica che la connessione non deve restare persistente.

`keep-alive` indica che la connessione può restare aperta per ulteriori richieste.

Nel contesto della lezione:
- in HTTP/1.1 le connessioni persistenti sono il comportamento previsto;
- in HTTP/1.0 `keep-alive` può essere usato per richiedere una forma di persistenza.

---

### **10. Altri Header di Richiesta**

La lezione cita alcuni header ulteriori.

#### **10.1. Referer**

Il campo `Referer` specifica da dove proviene l'utente.

È utile per:
- log;
- statistiche;
- tracciamento del percorso dell'utente;
- debugging;
- analisi del traffico.

Il nome `Referer` contiene storicamente un errore di spelling, ma è rimasto così nello standard e nell'uso pratico.

#### **10.2. From**

Il campo `From` può specificare l'indirizzo e-mail dell'utente.

In pratica è poco usato, anche per motivi di privacy.

#### **10.3. Authorization**

Il campo `Authorization` può inviare credenziali o informazioni di autenticazione per accedere a risorse protette.

Nel corso verrà ripreso in relazione alla configurazione pratica del web server Apache.

È un campo delicato, perché può trasportare dati sensibili.

#### **10.4. If-Modified-Since**

Il campo `If-Modified-Since` è usato con la cache del browser.

Il browser può salvare una copia locale di una pagina.

Quando richiede di nuovo la stessa risorsa, può chiedere al server:

> inviami il documento solo se è stato modificato dopo questa data.

Se il documento non è cambiato, il server può evitare di reinviarlo.

Questo riduce traffico e tempo di risposta.

> ⚠️ Alcuni header HTTP trasportano informazioni sensibili o traccianti. Anche se HTTP è semplice, i suoi header hanno implicazioni di privacy e sicurezza.

### **11. Formato della HTTP Response**

La risposta HTTP ha una struttura simile alla richiesta.

È composta da:
- status line;
- header di risposta;
- riga vuota;
- corpo, spesso contenente l'oggetto richiesto.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **11.1. Status Line**

La prima riga è la **status line**.

È sempre presente.

Contiene:
- versione del protocollo;
- codice di stato;
- messaggio di stato.

Forma generale:

```http
HTTP/versione codice messaggio
```

Esempio:

```http
HTTP/1.1 200 OK
```

#### **11.2. Header e Corpo**

Dopo la status line vengono gli header.

Poi c'è una riga vuota.

Dopo la riga vuota si trova il corpo della risposta, se presente.

Il corpo può contenere:
- file HTML;
- immagine;
- script;
- dati JSON;
- file scaricabile;
- messaggio di errore;
- contenuto generato dinamicamente.

### **12. Codici di Stato HTTP**

I codici di stato sono rappresentati da tre cifre.

La prima cifra indica la famiglia del codice.

| Prima cifra | Significato generale |
|-------------|----------------------|
| 1xx | Informativo |
| 2xx | Successo |
| 3xx | Redirezione o azione ulteriore |
| 4xx | Errore del client |
| 5xx | Errore del server |

Nel transcript vengono evidenziate soprattutto le famiglie 1, 2, 3, 4 e alcuni esempi della famiglia 5.

#### **12.1. Codici Informativi 1xx**

I codici `1xx` sono informativi.

Non erano previsti nella versione HTTP/1.0.

Indicano una risposta intermedia o informazione temporanea.

#### **12.2. Codici di Successo 2xx**

I codici `2xx` indicano che la richiesta ha avuto esito positivo.

Il più comune è:

```http
200 OK
```

Significa che la richiesta è stata gestita correttamente.

#### **12.3. Codici di Redirezione 3xx**

I codici `3xx` indicano che il client deve eseguire altre azioni perché la richiesta vada a buon fine.

Esempi:

```http
301 Moved Permanently
```

Il documento è stato spostato permanentemente.

```http
304 Not Modified
```

Il documento non è stato modificato, quindi il browser può usare la copia in cache.

#### **12.4. Errori del Client 4xx**

I codici `4xx` indicano errori lato client.

Esempi:

```http
400 Bad Request
```

Errore di sintassi nella richiesta.

```http
401 Unauthorized
```

È richiesta una fase di autenticazione.

```http
403 Forbidden
```

Il client non ha il permesso di accedere alla risorsa.

```http
404 Not Found
```

Il file o la risorsa non è stata trovata.

#### **12.5. Errori del Server 5xx**

I codici `5xx` indicano errori lato server.

Nel transcript viene citato:

```http
505 HTTP Version Not Supported
```

La versione del protocollo non è supportata dal server.

> 📌 I codici HTTP non sono solo messaggi per l'utente: guidano il comportamento del browser, la cache, le redirezioni e i meccanismi di autenticazione.

### **13. Header della HTTP Response**

La lezione cita alcuni header di risposta.

#### **13.1. Date**

Il campo `Date` specifica quando è stata creata la risposta.

#### **13.2. Server**

Il campo `Server` contiene informazioni identificative sul web server.

Può indicare, per esempio, software e versione.

Dal punto di vista della sicurezza, fornire troppe informazioni sul server può aiutare un attaccante a cercare vulnerabilità specifiche.

#### **13.3. Last-Modified**

Il campo `Last-Modified` indica quando la risorsa è stata modificata per l'ultima volta.

È utile per la gestione della cache.

#### **13.4. Content-Length**

Il campo `Content-Length` specifica la dimensione del contenuto in byte.

#### **13.5. Content-Type**

Il campo `Content-Type` specifica il formato del contenuto.

Esempi:
- `text/html`;
- `image/jpeg`;
- `application/json`;
- `text/plain`.

#### **13.6. Expires**

Il campo `Expires` indica fino a quando il browser può considerare valida una copia in cache.

Dopo la data di scadenza, il browser dovrebbe richiedere nuovamente la risorsa o verificarne la validità.

### **14. Metodi HTTP**

La request line contiene un metodo.

Il metodo specifica l'operazione richiesta al server.

I metodi principali trattati nella lezione sono:
- GET;
- HEAD;
- POST.

Questi sono supportati sia da HTTP/1.0 sia da HTTP/1.1.

#### **14.1. Metodo GET**

Il metodo `GET` richiede la lettura di una pagina o risorsa web.

Esempio:

```http
GET /index.html HTTP/1.1
```

GET può inviare parametri al server tramite l'URL, nella parte chiamata **query string**.

La query string è separata dal resto dell'URL dal simbolo:

$$?$$

ed è formata da coppie:

$$nome = valore$$

separate dal simbolo:

$$\&$$

Esempio:

```text
/pagina?nome1=test1&nome2=test2
```

Altro esempio:

```text
/login?name=John&password=john5
```

#### **14.2. Query String e Form**

La query string può essere usata per passare al server i valori dei campi di un form.

Dal punto di vista della sicurezza, questo è delicato perché:
- i parametri sono visibili nell'URL;
- possono finire nella cronologia del browser;
- possono comparire nei log;
- possono essere copiati o condivisi;
- possono essere manipolati dall'utente.

Per questo non è opportuno inserire dati sensibili in una query string.

Nel contesto della lezione viene sottolineato anche che i parametri passati via GET sono limitati, perché devono stare nell'URL.

#### **14.3. Metodo HEAD**

Il metodo `HEAD` è simile a `GET`, ma richiede solo l'intestazione della risposta.

Il server non invia il corpo dell'oggetto richiesto.

È utile per:
- controllare se una risorsa esiste;
- leggere metadata;
- verificare dimensione o data di modifica;
- controllare la cache.

#### **14.4. Metodo POST**

Il metodo `POST` invia informazioni nel corpo della richiesta, non direttamente nell'URL.

È usato spesso per:
- form;
- login;
- invio di dati;
- richieste che modificano stato lato server;
- upload o invio di contenuti.

Rispetto a GET, POST evita che i parametri siano visibili direttamente nell'URL.

Questo però non significa che POST sia sicuro se la comunicazione avviene in chiaro.

> ⚠️ POST nasconde i parametri dall'URL, ma non cifra il contenuto. Senza HTTPS, anche il corpo della richiesta può essere intercettato.

#### **14.5. Metodi di HTTP/1.1**

HTTP/1.1 introduce o standardizza ulteriori metodi, tra cui:
- DELETE;
- CONNECT;
- OPTIONS.

La lezione li cita per completezza, rimandando i dettagli al corso di Reti.

Per questo modulo sono meno centrali rispetto a GET, HEAD e POST.

### **15. Vulnerabilità e Limiti di Sicurezza di HTTP**

Dopo aver visto il funzionamento di HTTP, possiamo analizzarne i limiti di sicurezza.

HTTP prevede alcune forme di autenticazione, ma nella forma base resta debole.

In particolare:
- l'autenticazione HTTP è limitata;
- le password possono essere esposte se non protette;
- il traffico viene trasmesso in chiaro;
- non viene garantita la confidenzialità;
- non viene garantita l'integrità forte;
- il server non viene autenticato in modo robusto;
- un attaccante sul canale può osservare o alterare messaggi.

#### **15.1. Autenticazione Debole**

Guardando i codici di stato, si vede che HTTP prevede meccanismi di autenticazione.

Per esempio:

```http
401 Unauthorized
```

indica che è necessaria una fase di autenticazione.

Tuttavia, come verrà mostrato nella sessione pratica relativa ad Apache, questa forma di autenticazione è debole se usata senza ulteriori protezioni.

#### **15.2. Mancanza di Confidenzialità**

HTTP trasmette dati in chiaro.

Un attaccante che osserva il traffico può leggere:
- URL richiesti;
- parametri GET;
- header;
- cookie;
- eventuali credenziali;
- corpo delle richieste POST;
- contenuto delle risposte.

#### **15.3. Mancanza di Integrità**

HTTP non protegge in modo crittografico il contenuto dei messaggi.

Un attaccante attivo può potenzialmente:
- modificare una richiesta;
- modificare una risposta;
- iniettare contenuti;
- alterare script;
- cambiare risorse scaricate dal browser;
- manipolare redirezioni.

> 📌 Il problema principale di HTTP non è il formato del protocollo, ma il fatto che request e response viaggiano senza protezione crittografica.

### **16. HTTPS**

L'alternativa pratica a HTTP non sicuro è **HTTPS**, cioè **HTTP Secure**.

HTTPS non è un protocollo completamente nuovo.

È HTTP eseguito sopra un canale sicuro, basato su SSL o TLS.

Le HTTP request e le HTTP response restano concettualmente le stesse.

La differenza è che vengono trasmesse dentro una connessione protetta.

#### **16.1. HTTP su TLS**

Nel caso di HTTP:

$$HTTP \rightarrow TCP$$

Nel caso di HTTPS:

$$HTTP \rightarrow TLS/SSL \rightarrow TCP$$

Il browser non apre semplicemente una connessione TCP in chiaro.

Stabilisce prima una connessione sicura usando TLS.

Poi invia i messaggi HTTP dentro quella connessione.

#### **16.2. Porta di HTTPS**

HTTP usa tipicamente la porta:

$$80$$

HTTPS usa tipicamente la porta:

$$443$$

#### **16.3. Origine Storica**

HTTPS nasce nel 1994 in ambito Netscape Communications, per supportare comunicazioni sicure nel browser Netscape Navigator.

La motivazione pratica era rendere possibile il commercio elettronico sicuro.

#### **16.4. Autenticazione del Server**

HTTPS usa un certificato digitale in formato X.509 per autenticare il server.

Il browser verifica il certificato del server per capire se sta comunicando con il sito corretto.

Su richiesta può essere autenticato anche il client, ma nella navigazione web comune l'autenticazione tramite certificato client è molto meno frequente.

#### **16.5. Cifratura Efficiente**

Anche se HTTPS usa certificati e quindi crittografia asimmetrica nella fase iniziale, i dati HTTP non vengono normalmente cifrati tutti con crittografia asimmetrica.

Durante la fase di handshake, browser e server concordano una chiave di sessione.

Poi request e response vengono protette con crittografia simmetrica.

Questo rende il protocollo efficiente.

> ✅ HTTPS combina autenticazione tramite certificato e cifratura efficiente tramite chiavi di sessione.

### **18. S-HTTP**

Storicamente è esistita anche un'altra proposta: **S-HTTP**, cioè **Secure HTTP**.

S-HTTP è descritto nella RFC 2660 ed è stato proposto nel 1999.

A differenza di HTTPS, S-HTTP modificava direttamente il protocollo HTTP.

Supportava diversi algoritmi crittografici per la cifratura e la protezione dei messaggi.

Tuttavia non è diventato uno standard di fatto e non è supportato universalmente dai browser.

Per questo, nella pratica, la soluzione usata è HTTPS.

#### **18.1. Perché HTTPS ha Prevalso**

HTTPS ha avuto maggiore successo perché:
- mantiene HTTP sostanzialmente invariato;
- inserisce la protezione sotto HTTP;
- usa TLS/SSL come livello sicuro;
- è più facilmente integrabile nei browser e nei server;
- è diventato interoperabile e supportato diffusamente.

> 📌 HTTPS protegge HTTP senza cambiare radicalmente il formato delle request e delle response; S-HTTP modificava HTTP e non si è imposto nella pratica.

### **20. Riepilogo**

> ✅ HTTP è un protocollo applicativo client-server usato per trasferire oggetti web tra browser e web server. Si appoggia su TCP, usa tipicamente la porta 80 ed è stateless. HTTP/1.0 usa connessioni non persistenti, mentre HTTP/1.1 supporta connessioni persistenti per ridurre overhead. Le request sono composte da request line, header, riga vuota e corpo opzionale; le response contengono status line, header, riga vuota e corpo. I metodi principali sono GET, HEAD e POST; i codici di stato indicano successo, redirezione, errori client e server. Dal punto di vista della sicurezza, HTTP trasmette dati in chiaro e offre autenticazione debole: non garantisce confidenzialità né integrità. HTTPS risolve questi problemi eseguendo HTTP sopra TLS/SSL, usando certificati X.509 per autenticare il server e chiavi di sessione simmetriche per cifrare efficientemente request e response.
