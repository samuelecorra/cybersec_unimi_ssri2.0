# L1 — Introduzione al Web e alla Sicurezza delle Applicazioni Web

> 📌 La sicurezza delle applicazioni web riguarda l'intero ecosistema: browser, web server, database, protocolli di comunicazione, linguaggi di scripting e logica applicativa. Il web è distribuito, dinamico e usato da miliardi di utenti: per questo le vulnerabilità web hanno un impatto enorme.

---

### **1. Obiettivo della Lezione**

In questa lezione viene descritto brevemente il funzionamento del web, evidenziando le principali aree di vulnerabilità che verranno approfondite nel modulo.

L'obiettivo non è ripetere in modo completo il funzionamento dei sistemi web, già trattato in altri corsi, ma fissare il contesto di sicurezza:
- quali attori partecipano a un'applicazione web;
- quali tecnologie permettono la comunicazione;
- come una pagina viene richiesta e restituita;
- che cosa identifica una risorsa web;
- perché i siti dinamici aumentano la superficie di attacco;
- quali classi di vulnerabilità sono più frequenti.

La lezione introduce quindi il passaggio dal tema dei protocolli crittografici alla sicurezza concreta delle applicazioni web.

> 💡 Nei protocolli crittografici il focus era proteggere i messaggi; nelle applicazioni web bisogna proteggere anche codice, dati, sessioni, browser, server e database.

### **2. Il Web come Applicazione di Internet**

I sistemi web sono applicazioni particolari di Internet.

Internet è un mezzo di comunicazione:
- globale;
- interattivo;
- multimediale;
- ipertestuale.

Ha cambiato radicalmente il modo di comunicare, lavorare, distribuire contenuti, accedere a servizi e gestire informazioni.

Le applicazioni web sono usate ogni giorno da un numero enorme di utenti per attività molto diverse:
- informazione;
- posta elettronica;
- home banking;
- e-commerce;
- social network;
- servizi pubblici;
- gestione documentale;
- applicazioni aziendali;
- piattaforme didattiche;
- servizi sanitari.

Un errore in un'applicazione web può esporre dati personali, credenziali, pagamenti, documenti riservati o funzioni amministrative.

> 📌 Il web non è solo pubblicazione di pagine: oggi è una piattaforma applicativa completa, usata anche per operazioni critiche e dati sensibili.

### **3. Idea di Base del Web**

L'idea alla base del web è archiviare pagine di ipertesto su computer collegati tra loro tramite Internet.

Queste pagine possono essere collegate tra loro tramite link, indipendentemente dalla loro collocazione fisica.

Un utente può accedere a una risorsa da qualunque computer collegato a Internet, specificando un nome simbolico.

Questo nome simbolico è l'**URL**, cioè **Uniform Resource Locator**.

#### **3.1. Ipertesto**

Un ipertesto è un documento che contiene collegamenti verso altri documenti o risorse.

Il web nasce proprio da questa idea:
- una pagina può contenere link verso altre pagine;
- i link possono puntare a risorse sullo stesso server;
- i link possono puntare a risorse su server diversi;
- l'utente naviga seguendo collegamenti logici, non la posizione fisica dei file.

#### **3.2. Distribuzione Fisica delle Risorse**

Le risorse web non devono trovarsi tutte nello stesso luogo.

Una pagina può collegare contenuti:
- ospitati sul server principale;
- caricati da repository separati;
- distribuiti da server geograficamente lontani;
- generati dinamicamente da database;
- caricati da servizi esterni.

Questo rende il web flessibile, ma aumenta anche la complessità di sicurezza.

> ⚠️ Una pagina web moderna può dipendere da molte risorse esterne. Ogni dipendenza può diventare un punto di attacco o di esposizione.

### **4. Attori Coinvolti in un'Applicazione Web**

Per discutere la sicurezza dei sistemi web bisogna identificare gli attori principali.

In un'applicazione web tipica troviamo:
- browser;
- web server;
- database lato server;
- Internet come canale di comunicazione.

#### **4.1. Browser**

Il **browser** è il programma applicativo che gira sul computer dell'utente.

Il suo ruolo è:
- ricevere l'URL digitato o selezionato dall'utente;
- inviare richieste HTTP;
- ricevere risposte dal server;
- interpretare HTML, CSS e JavaScript;
- visualizzare la pagina;
- gestire cookie, sessioni, cache e credenziali;
- eseguire codice lato client.

Dal punto di vista della sicurezza, il browser è importante perché esegue codice proveniente dal web e gestisce dati dell'utente.

#### **4.2. Web Server**

Il **web server** è il fornitore dei servizi e delle risorse.

Il suo ruolo è:
- ricevere richieste dai browser;
- interpretare l'URL richiesto;
- recuperare o generare la risorsa;
- interagire con database o altri servizi;
- inviare una risposta HTTP al browser.

Il web server può servire file statici oppure invocare logica applicativa per generare contenuti dinamici.

#### **4.3. Database Lato Server**

Molte applicazioni web usano una base di dati lato server.

Il database può contenere:
- pagine web;
- dati usati per generare pagine;
- account utente;
- password o hash di password;
- profili;
- ordini;
- pagamenti;
- autorizzazioni;
- log applicativi;
- dati confidenziali.

Il web server interagisce con il database per recuperare, modificare o cancellare dati.

Questa interazione è uno dei punti più delicati della sicurezza web.

#### **4.4. Internet come Mezzo di Comunicazione**

Internet collega browser e server.

È un canale di comunicazione che può essere attraversato da attaccanti.

Per questo bisogna considerare:
- intercettazione;
- modifica;
- ritardo;
- cancellazione;
- replay;
- impersonificazione;
- attacchi ai protocolli.

> 📌 Un'applicazione web sicura deve proteggere sia gli endpoint sia la comunicazione tra endpoint.

### **5. Tecnologie di Base del Web**

Il web si basa su più tecnologie combinate.

Le più rilevanti per questa lezione sono:
- HTTP;
- suite TCP/IP;
- HTML;
- linguaggi di scripting;
- database e tecnologie lato server.

#### **5.1. HTTP e TCP/IP**

Il protocollo HTTP e la suite TCP/IP permettono la comunicazione tra browser e server.

La comunicazione segue in genere il paradigma **richiesta-risposta**:
- il server resta in attesa di richieste;
- il browser invia una richiesta;
- il server elabora la richiesta;
- il server invia una risposta.

Questo schema verrà approfondito nella lezione successiva.

#### **5.2. HTML e Ipertesto**

HTML è il linguaggio usato per descrivere la struttura delle pagine web.

Una pagina HTML può contenere:
- testo;
- link;
- immagini;
- form;
- riferimenti a script;
- riferimenti a fogli di stile;
- oggetti multimediali;
- contenuti caricati da altri server.

#### **5.3. Linguaggi di Scripting**

Le applicazioni web moderne usano linguaggi di scripting e tecnologie dinamiche.

Esempi citati nel contesto della lezione:
- PHP;
- JavaScript;
- tecnologie lato server equivalenti.

Gli script permettono di:
- generare pagine dinamiche;
- leggere input utente;
- interrogare database;
- gestire sessioni;
- personalizzare contenuti;
- modificare l'interfaccia lato client.

> ⚠️ La potenza dei linguaggi di scripting aumenta anche la superficie di attacco: input non validato, script iniettati, sessioni gestite male e query costruite in modo insicuro sono cause frequenti di vulnerabilità.

### **6. Esempio di Richiesta di una Pagina Web**

Consideriamo un utente che vuole visitare un sito, per esempio un sito di notizie.

L'utente inserisce nel browser un URL.

Quando preme invio:
- il browser interpreta l'URL;
- individua il server a cui inviare la richiesta;
- invia una richiesta HTTP;
- il server riceve la richiesta;
- il server recupera o genera la pagina;
- il server invia una risposta HTTP;
- il browser visualizza la pagina ricevuta.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **6.1. Richiesta HTTP**

La richiesta HTTP è il messaggio inviato dal browser al server.

Serve a dire:
- quale risorsa viene richiesta;
- quale metodo viene usato;
- quali intestazioni accompagnano la richiesta;
- quali dati eventuali vengono inviati.

Nel caso più semplice, il browser chiede una pagina HTML.

#### **6.2. Risposta HTTP**

La risposta HTTP è il messaggio inviato dal server al browser.

Può contenere:
- codice di stato;
- intestazioni;
- pagina HTML;
- dati;
- redirect;
- errori;
- contenuti generati dinamicamente.

#### **6.3. Pagina Composta da Più Oggetti**

Una pagina HTML non è necessariamente un oggetto unico.

Può riferirsi a molte risorse:
- immagini;
- fogli di stile;
- script;
- dati meteo;
- notizie;
- pubblicità;
- contenuti multimediali;
- componenti provenienti da server diversi.

Questi oggetti possono trovarsi su server differenti, anche collocati in aree geografiche diverse.

> 💡 Quando il browser carica una pagina, spesso avvia molte richieste aggiuntive per recuperare gli oggetti collegati alla pagina principale.

### **7. URL: Uniform Resource Locator**

L'**URL** è il meccanismo usato per identificare in modo univoco una risorsa web.

Ogni pagina o risorsa viene identificata specificando almeno:
- il protocollo usato per lo scambio;
- il nome del computer o dominio che ospita la risorsa;
- il nome o percorso del file all'interno di quel computer.

Una forma tipica è:

$$protocollo://nome\_dominio/percorso/risorsa$$

Per esempio:

$$http://www.example.com/index.html$$

#### **7.1. Componenti dell'URL**

Le componenti principali sono:
- **schema o protocollo**, per esempio HTTP o HTTPS;
- **nome di dominio**, che identifica simbolicamente il server;
- **percorso**, che identifica la risorsa sul server;
- eventuali parametri, query string o frammenti.

Dal punto di vista della sicurezza, l'URL è importante perché può contenere input controllato dall'utente.

Per esempio, parametri nella query string possono influenzare:
- quale contenuto viene mostrato;
- quale record viene letto dal database;
- quale azione viene eseguita;
- quale sessione o token viene usato.

> ⚠️ Un URL non è solo un indirizzo: può trasportare dati che l'applicazione userà per prendere decisioni.

### **8. Siti Statici e Siti Dinamici**

Esistono due grandi categorie di siti web:
- siti statici;
- siti dinamici.

#### **8.1. Siti Statici**

I siti statici appartengono al web storico.

Presentano contenuti recuperati da pagine web statiche.

In genere:
- i file HTML sono già pronti;
- i contenuti cambiano poco frequentemente;
- le modifiche vengono fatte manualmente;
- una o più persone aggiornano direttamente il codice della pagina;
- possono essere usati editor web per modificare i file.

La superficie di attacco può essere più limitata, perché il server si limita spesso a restituire file.

Questo non significa che un sito statico sia automaticamente sicuro, ma la logica applicativa è ridotta.

#### **8.2. Siti Dinamici**

I siti dinamici generano contenuti in modo automatico.

Il contenuto può variare in base a:
- utente autenticato;
- parametri della richiesta;
- dati nel database;
- preferenze;
- posizione;
- orario;
- stato della sessione;
- azioni precedenti;
- autorizzazioni.

Il web dinamico è il caso più comune nelle applicazioni contemporanee.

Esempi:
- area personale di una banca;
- carrello di un e-commerce;
- social network;
- portali universitari;
- posta elettronica web;
- dashboard amministrative.

#### **8.3. Impatto sulla Sicurezza**

I siti dinamici aumentano la complessità perché combinano:
- input dell'utente;
- logica applicativa;
- sessioni;
- database;
- scripting lato client;
- scripting lato server;
- autorizzazioni;
- generazione dinamica di HTML.

Questa complessità introduce molte possibili vulnerabilità.

> 📌 Le vulnerabilità web moderne sono spesso vulnerabilità del web dinamico: input e stato applicativo vengono usati per generare risposte personalizzate.

### **10. Che Cosa Significa Rendere Sicura un'Applicazione Web**

Rendere sicura un'applicazione web non significa proteggere un solo componente.

Bisogna considerare più livelli.

#### **10.1. Sicurezza Lato Client**

La sicurezza lato client riguarda il browser e l'ambiente dell'utente.

Include:
- esecuzione sicura di JavaScript;
- isolamento tra siti;
- gestione dei cookie;
- protezione delle credenziali;
- protezione contro script malevoli;
- prevenzione di furto di sessione;
- attenzione a contenuti caricati da terze parti.

#### **10.2. Sicurezza del Browser**

Il browser è un componente complesso.

Deve:
- interpretare HTML;
- eseguire JavaScript;
- applicare policy di sicurezza;
- gestire certificati;
- gestire cookie e storage locale;
- separare origini diverse;
- proteggere l'utente da contenuti pericolosi.

Un attacco web può quindi colpire anche il comportamento del browser.

#### **10.3. Sicurezza Lato Server**

La sicurezza lato server riguarda:
- web server;
- framework applicativi;
- codice lato server;
- gestione utenti;
- autorizzazioni;
- sessioni;
- accesso a file;
- configurazioni;
- logging;
- aggiornamenti e patch.

Il server è il punto che gestisce risorse e dati.

Se viene compromesso, l'impatto può essere molto elevato.

#### **10.4. Sicurezza dei Protocolli di Comunicazione**

Browser e server comunicano tramite protocolli.

Bisogna garantire:
- confidenzialità;
- integrità;
- autenticazione del server;
- protezione contro intercettazione;
- protezione contro modifica;
- gestione corretta dei certificati;
- uso corretto di HTTP e HTTPS.

Le lezioni successive approfondiranno questi aspetti.

#### **10.5. Sicurezza dell'Interazione con il DBMS**

Il web server spesso interagisce con un database management system.

Questa interazione deve essere sicura perché il database contiene dati critici.

Problemi tipici:
- query costruite con input non validato;
- autorizzazioni eccessive;
- esposizione di errori SQL;
- mancato controllo dell'identità utente;
- accesso a dati di altri utenti;
- injection.

#### **10.6. Sicurezza del Web Dinamico**

Il web dinamico richiede attenzione all'uso dei linguaggi di scripting.

Bisogna controllare:
- input utente;
- output generato;
- escape dei dati;
- gestione delle sessioni;
- autenticazione;
- autorizzazione;
- interazione con database;
- codice lato client;
- codice lato server.

> 📌 La sicurezza web è una proprietà end-to-end: un solo componente debole può compromettere l'intera applicazione.

#### **11.2. Vulnerabilità Ricorrenti**

Tra le vulnerabilità più comuni citate nella lezione ci sono:
- injection;
- problemi di autenticazione;
- cattiva gestione della sessione;
- Cross-Site Scripting.

Queste categorie saranno studiate più in dettaglio nelle lezioni successive.

#### **11.3. Injection**

Le vulnerabilità di tipo **injection** si verificano quando input controllato dall'utente viene interpretato come comando o parte di un linguaggio.

L'esempio più classico è SQL injection.

In quel caso l'attaccante manipola l'input in modo da modificare la query eseguita dal server sul database.

#### **11.4. Autenticazione e Gestione della Sessione**

Le vulnerabilità di autenticazione e session management riguardano:
- login;
- password;
- token di sessione;
- cookie;
- logout;
- scadenza delle sessioni;
- recupero account;
- protezione contro impersonificazione.

Una cattiva gestione della sessione può permettere a Trudy di agire come un utente legittimo.

#### **11.5. Cross-Site Scripting**

Il **Cross-Site Scripting** riguarda l'inserimento di script malevoli in pagine viste da altri utenti.

L'attacco sfrutta il fatto che il browser esegue codice ricevuto dal sito.

Se l'applicazione include input non sanitizzato dentro la pagina, Trudy può far eseguire JavaScript nel browser della vittima.

La lezione richiama anche statistiche prodotte da altre fonti.

Anche in quelle statistiche compaiono con alta frequenza attacchi come:
- SQL injection;
- Cross-Site Scripting.

### **13. Riepilogo**

> ✅ Il web è un'applicazione di Internet basata su ipertesti distribuiti, URL, browser, web server, HTTP, TCP/IP, HTML, scripting e database. Una pagina web viene richiesta dal browser tramite HTTP e può essere composta da molti oggetti provenienti anche da server diversi. I siti statici servono pagine predefinite, mentre i siti dinamici generano contenuti in base a input, sessioni e dati nel database. La sicurezza web è importante per la scala enorme di siti e utenti e per la quantità di dati confidenziali gestiti. Rendere sicura un'applicazione web significa proteggere client, browser, server, protocolli, database e codice dinamico. Le vulnerabilità ricorrenti includono injection, problemi di autenticazione/sessione e Cross-Site Scripting, che verranno approfonditi nelle lezioni successive.
