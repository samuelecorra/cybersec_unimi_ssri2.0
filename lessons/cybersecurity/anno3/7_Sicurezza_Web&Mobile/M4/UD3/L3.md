# L3 — SQL Injection: Contromisure

> 📌 La difesa contro la SQL injection si basa su un principio centrale: **i dati forniti dall'utente non devono mai poter diventare elementi di controllo della query SQL**. Per ottenere questo risultato servono validazione dell'input, uso corretto di prepared statement e bind variable tipate.

---

### **1. Obiettivo della Lezione**

In questa lezione vengono introdotte le principali contromisure agli attacchi di tipo **SQL injection**.

Nella lezione precedente è stato visto che l'attacco funziona quando:
- il client invia input non fidato;
- il server non controlla quell'input;
- lo script lato server costruisce una query SQL concatenando stringhe;
- l'input dell'utente diventa parte della sintassi SQL;
- la semantica della query viene alterata.

Questa lezione studia come ridurre o eliminare questa vulnerabilità.

Le contromisure principali sono:
- validazione dell'input;
- uso di blacklist;
- uso di whitelist;
- uso di espressioni regolari;
- uso di **prepared statement**;
- uso di **bind variable** tipate;
- separazione rigorosa tra dati e controllo.

> ⚠️ Nessuna singola tecnica va considerata una soluzione universale. La difesa robusta nasce dall'uso coerente di più contromisure.

---

### **3. Dati ed Elementi di Controllo**

Il punto concettuale fondamentale è la distinzione tra:
- **dati**;
- **elementi di controllo**.

Un dato è qualcosa che l'applicazione dovrebbe memorizzare o confrontare.

Esempi:
- `alice`;
- `mario.rossi`;
- `O'Brian`;
- `42`;
- `2026-06-26`.

Un elemento di controllo è invece una parte della sintassi SQL che modifica il comportamento della query.

Esempi:
- apici;
- punto e virgola;
- commenti;
- operatori booleani;
- parole chiave SQL;
- nomi di tabelle;
- nomi di attributi;
- comandi come `DROP`, `INSERT`, `UPDATE`, `DELETE`.

In una query sicura, i dati restano dati.

In una query vulnerabile, l'attaccante riesce a inserire elementi di controllo dove il server si aspettava semplici dati.

> 📌 La SQL injection è una violazione del confine tra dato e codice.

---

### **4. Il Ruolo dei Metacaratteri**

Gli attacchi di SQL injection sfruttano spesso **metacaratteri**, cioè caratteri che hanno un significato speciale nel linguaggio SQL.

Esempi importanti:
- apice singolo `'`;
- virgolette doppie `"`, a seconda del DBMS e del contesto;
- punto e virgola `;`;
- doppio trattino `--`;
- parentesi;
- spazi bianchi;
- operatori come `=`, `<`, `>`.

L'apice è particolarmente importante perché separa le stringhe SQL.

Per esempio:

```sql
WHERE username = 'alice'
```

Se l'input utente viene inserito tra apici, un apice malevolo può chiudere la stringa prima del previsto.

Da lì l'attaccante può aggiungere condizioni o comandi.

Esempio:

```sql
' OR 1=1 --
```

In questo caso:
- l'apice chiude la stringa;
- `OR 1=1` aggiunge una condizione sempre vera;
- `--` commenta la parte rimanente della query.

> ⚠️ I metacaratteri sono pericolosi quando l'applicazione li lascia arrivare direttamente alla query come parte della sintassi SQL.

---

### **5. Prima Contromisura: Validazione dell'Input**

La contromisura più immediata è validare l'input ricevuto dal client.

Validare significa controllare che il dato ricevuto:
- abbia il tipo atteso;
- abbia il formato atteso;
- abbia lunghezza ragionevole;
- appartenga a un insieme di valori ammessi;
- non contenga elementi incompatibili con il contesto.

Esempi:
- un identificativo numerico deve contenere solo cifre;
- un mese deve essere compreso tra 1 e 12;
- una data deve rispettare il formato previsto;
- un codice fiscale deve rispettare il pattern atteso;
- un campo username deve avere solo caratteri ammessi;
- un campo prezzo deve essere numerico.

La validazione deve avvenire lato server.

I controlli lato client possono migliorare l'esperienza utente, ma non sono una misura di sicurezza sufficiente.

> ⚠️ Il client non è affidabile: l'attaccante può bypassare JavaScript, modificare il form o costruire direttamente richieste HTTP.

---

### **6. Blacklisting**

Un primo approccio alla validazione è il **blacklisting**.

Con il blacklisting si definisce una lista di caratteri, stringhe o pattern vietati.

L'input viene rifiutato o filtrato se contiene elementi presenti nella lista nera.

Nel transcript vengono citati esempi come:
- apici;
- spazi bianchi;
- punti e virgola;
- trattini;
- comandi frequenti come `DROP TABLE`;
- comandi frequenti come `DROP DATABASE`.

Esempio concettuale:

```text
Rifiuta input che contiene:
'
;
--
DROP TABLE
DROP DATABASE
```

Questo approccio prova a bloccare direttamente gli elementi usati negli attacchi.

#### **6.1. Vantaggi del Blacklisting**

Il blacklisting è intuitivo.

Permette di dire:

> questi caratteri o comandi non devono comparire nell'input.

Può essere utile per intercettare tentativi evidenti di attacco.

Per esempio, un campo che dovrebbe contenere solo un numero non dovrebbe contenere:
- `DROP`;
- `UNION`;
- apici;
- commenti SQL;
- punto e virgola.

#### **6.2. Limiti del Blacklisting**

Il limite principale è che bisogna conoscere in anticipo tutto ciò che è pericoloso.

Questo è difficile perché:
- si può dimenticare un carattere;
- si può dimenticare una parola chiave;
- esistono varianti sintattiche tra DBMS diversi;
- l'attaccante può usare encoding alternativi;
- l'attaccante può aggirare controlli troppo semplici;
- alcune parole vietate possono comparire legittimamente in dati reali;
- un filtro incompleto può dare falsa sicurezza.

> ⚠️ Con il blacklisting non si ha mai la certezza di aver elencato tutti i casi pericolosi.

#### **6.3. Conflitto con i Requisiti Funzionali**

Un altro problema è il conflitto con i requisiti funzionali.

Nel transcript viene fatto l'esempio del cognome:

```text
O'Brian
```

Se l'applicazione vieta sempre gli apici, non può memorizzare correttamente un cognome che contiene un apice.

Il carattere `'` è pericoloso in SQL se viene interpretato come controllo, ma può essere perfettamente legittimo come dato.

Questo mostra che filtrare caratteri in modo cieco può rendere l'applicazione meno corretta.

> 💡 Il problema non è che `O'Brian` contenga un apice; il problema è trattare quell'apice come sintassi SQL invece che come parte del dato.

#### **6.4. Blacklisting come Misura di Supporto**

Il blacklisting non è una panacea di sicurezza.

Può essere usato come misura aggiuntiva, ma non dovrebbe essere l'unico meccanismo di difesa.

È più sensato usarlo in combinazione con:
- whitelist;
- validazione di tipo;
- prepared statement;
- bind variable;
- permessi minimi sul database;
- gestione corretta degli errori.

> ⚠️ Il blacklisting da solo è fragile: può bloccare input leciti e lasciar passare input malevoli non previsti.

---

### **7. Whitelisting**

L'approccio opposto è il **whitelisting**.

Con il whitelisting si definisce l'insieme degli input ammessi.

Tutto ciò che non appartiene a quell'insieme viene rifiutato.

Questa impostazione è più robusta perché parte da una domanda diversa.

Non chiede:

> quali input sono pericolosi?

Chiede invece:

> quali input sono validi per questa funzione?

Esempi:
- un ID utente deve essere composto solo da cifre;
- un mese deve essere un numero in un intervallo definito;
- una scelta da menu deve appartenere a un insieme finito di valori;
- un ordinamento può essere solo `asc` o `desc`;
- una categoria prodotto deve appartenere a categorie note;
- un campo booleano può essere solo `true` o `false`.

> 📌 Il whitelisting è preferibile perché definisce positivamente ciò che l'applicazione accetta.

---

### **8. Espressioni Regolari per la Whitelist**

L'insieme dei valori ammessi viene spesso definito con **espressioni regolari**.

Un'espressione regolare descrive un pattern.

L'input viene accettato solo se corrisponde al pattern.

Nel transcript viene fatto l'esempio di un parametro che deve essere un intero non negativo.

Un pattern compatibile con l'idea della lezione è:

```regex
^[0-9]*$
```

Significato:
- `^` indica l'inizio della stringa;
- `[0-9]` indica una cifra da 0 a 9;
- `*` indica zero o più ripetizioni;
- `$` indica la fine della stringa.

Quindi il pattern dice:

> dall'inizio alla fine della stringa devono comparire solo cifre.

#### **8.1. Osservazione sul Caso Vuoto**

Il simbolo `*` indica **zero o più** ripetizioni.

Quindi:

```regex
^[0-9]*$
```

accetta anche la stringa vuota.

Se il campo deve contenere almeno una cifra, è più adatto:

```regex
^[0-9]+$
```

dove `+` significa una o più ripetizioni.

> ⚠️ Questo è un dettaglio importante: una regex può essere sintatticamente corretta ma troppo permissiva rispetto al requisito applicativo.

#### **8.2. Esempio: Parametro Mese**

Se il requisito è solo:

> intero non negativo

un pattern come:

```regex
^[0-9]+$
```

è coerente.

Se invece il parametro rappresenta davvero un mese dell'anno, la validazione dovrebbe essere più restrittiva.

Per esempio:

```regex
^(0?[1-9]|1[0-2])$
```

Questo pattern accetta:
- `1`;
- `2`;
- ...
- `9`;
- `01`;
- `02`;
- ...
- `09`;
- `10`;
- `11`;
- `12`.

Non accetta:
- `0`;
- `13`;
- `99`;
- stringhe con lettere;
- stringhe con comandi SQL.

> 💡 La whitelist deve riflettere il significato reale del campo, non solo bloccare alcuni caratteri pericolosi.

---

### **9. Limiti della Sola Validazione**

La validazione dell'input è necessaria, ma non basta da sola.

Infatti:
- alcuni dati leciti possono contenere caratteri speciali;
- alcuni campi sono naturalmente testuali e difficili da restringere;
- la validazione può essere implementata male;
- possono esistere più punti di ingresso non validati;
- un requisito applicativo può cambiare nel tempo;
- il controllo può essere aggirato se non è centralizzato.

Esempio:

```text
O'Brian
```

è un dato valido.

Un'applicazione deve poterlo gestire senza permettere SQL injection.

Questo porta alla contromisura più importante: separare strutturalmente dati e comandi SQL.

---

### **10. Prepared Statement**

I **prepared statement** sono una delle contromisure fondamentali contro la SQL injection.

L'idea è costruire una query come **template statico**.

Il template contiene segnaposto per i valori, invece di concatenare direttamente l'input dell'utente.

Esempio:

```sql
SELECT *
FROM users
WHERE username = ?
AND password = ?;
```

I punti interrogativi `?` sono segnaposto.

Non sono testo inserito dall'utente.

Indicano al sistema:

> qui arriverà un dato, non una porzione di sintassi SQL.

Il database o il driver preparano la struttura della query separatamente dai valori.

Poi, in fase di esecuzione, i valori vengono associati ai segnaposto.

> 📌 Il prepared statement serve a mantenere separati codice SQL e dati dell'utente.

---

### **11. Bind Variable**

Le **bind variable** sono i parametri del prepared statement.

Nel template SQL sono rappresentate da segnaposto, tipicamente `?`.

In fase di esecuzione vengono sostituite con valori reali attraverso metodi specifici.

Esempio concettuale:

```java
String sql = "SELECT * FROM users WHERE username = ? AND password = ?";
PreparedStatement stmt = connection.prepareStatement(sql);

stmt.setString(1, username);
stmt.setString(2, password);

ResultSet rs = stmt.executeQuery();
```

Qui:
- il primo `?` è associato a `username`;
- il secondo `?` è associato a `password`;
- `setString` indica che i valori sono stringhe;
- l'input non viene concatenato dentro la query.

Il valore dell'utente viene trattato come dato anche se contiene caratteri speciali.

Per esempio, se `username` vale:

```sql
' OR 1=1 --
```

non diventa una condizione SQL.

Viene trattato come una stringa da confrontare con il campo `username`.

> 📌 La bind variable tipata impedisce all'input di cambiare la struttura sintattica della query.

---

### **12. Bind Variable Tipate**

Nel transcript viene sottolineato che le bind variable sono **tipate**.

Questo significa che il programmatore non inserisce genericamente testo nella query, ma usa metodi coerenti con il tipo atteso.

Esempi in stile Java:

```java
stmt.setString(1, username);
stmt.setString(2, password);
stmt.setInt(3, userId);
stmt.setDate(4, birthDate);
stmt.setBigDecimal(5, price);
```

Il tipo del valore viene controllato e convertito secondo le regole del driver.

Questo riduce il rischio che input arbitrario venga interpretato come sintassi SQL.

Esempio:

```java
stmt.setInt(1, userId);
```

Se il parametro deve essere un intero, il sistema non dovrebbe accettare una stringa come:

```sql
1 OR 1=1
```

come codice SQL.

> 💡 La tipizzazione è utile perché rafforza l'idea che ogni parametro abbia un ruolo preciso: stringa, intero, data, valore decimale, e così via.

### **13. Esempio Sicuro: Login con Prepared Statement**

Riprendiamo il form di login della lezione precedente.

Versione vulnerabile:

```java
String query =
    "SELECT * FROM users WHERE username = '" + username +
    "' AND password = '" + password + "'";
```

Versione corretta con prepared statement:

```java
String query =
    "SELECT * FROM users WHERE username = ? AND password = ?";

PreparedStatement statement = connection.prepareStatement(query);
statement.setString(1, username);
statement.setString(2, password);

ResultSet result = statement.executeQuery();
```

Nella versione sicura:
- la query SQL ha struttura fissa;
- username e password sono valori;
- i valori vengono associati ai placeholder;
- il driver sa che si tratta di dati;
- l'attaccante non può aggiungere `OR 1=1` come controllo SQL;
- l'attaccante non può commentare il resto della query con `--`.


il database cerca letteralmente un username uguale a quella stringa.

Non interpreta la stringa come frammento della clausola `WHERE`.

> ✅ Questa è la difesa strutturale: non "pulire meglio" una query concatenata, ma evitare che l'input diventi query.

### **14. Uso Ingenuo dei Prepared Statement**

Nel transcript viene evidenziato un punto importante:

> usare un prepared statement in modo ingenuo può lasciare l'applicazione vulnerabile.

Il problema si verifica quando il programmatore usa `PreparedStatement`, ma continua a costruire la query tramite concatenazione di stringhe.

Esempio vulnerabile:

```java
String query =
    "SELECT * FROM users WHERE username = '" + username +
    "' AND password = '" + password + "'";

PreparedStatement statement = connection.prepareStatement(query);
ResultSet result = statement.executeQuery();
```

A prima vista compare la parola `PreparedStatement`.

Ma la query è già stata costruita in modo insicuro prima della preparazione.

Il driver riceve una query completa che può già contenere il payload dell'attaccante.

Il problema non è risolto.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

> ⚠️ Un prepared statement senza bind variable non protegge dalla SQL injection.

#### **14.1. Regola Pratica**

La regola pratica è:

> non basta usare l'oggetto `PreparedStatement`; bisogna usare correttamente i placeholder e associare i valori con metodi tipati.

Forma corretta:

```java
String query = "SELECT * FROM users WHERE username = ? AND password = ?";
PreparedStatement statement = connection.prepareStatement(query);
statement.setString(1, username);
statement.setString(2, password);
```

Forma sospetta o vulnerabile:

```java
String query = "SELECT * FROM users WHERE username = '" + username + "'";
PreparedStatement statement = connection.prepareStatement(query);
```

Nel secondo caso, la concatenazione è ancora presente.

Quindi l'input può ancora alterare la query.

### **15. Separazione tra Controllo e Dati**

L'idea alla base di prepared statement e bind variable è che:

> dati ed elementi di controllo devono rimanere distinti.

Gli elementi di controllo sono fissati dal programmatore nel template SQL.

Esempio:

```sql
SELECT *
FROM users
WHERE username = ?
AND password = ?;
```

La struttura della query è:
- seleziona dalla tabella `users`;
- applica una condizione su `username`;
- applica una condizione su `password`.

L'utente può fornire solo i valori da confrontare.

Non può decidere:
- di aggiungere `OR 1=1`;
- di commentare la password;
- di aggiungere `DROP TABLE`;
- di inserire `UNION SELECT`;
- di cambiare la tabella interrogata.

> 📌 La query sicura non nasce dal fatto che l'input è "ben educato", ma dal fatto che l'input non ha più il potere di modificare la grammatica SQL.

---

### **16. Combinare Whitelist e Prepared Statement**

La soluzione consigliata dalla lezione è usare più tecniche in modo coerente.

In particolare:
- validare l'input con whitelist;
- usare prepared statement;
- usare bind variable tipate.

Queste tecniche coprono aspetti diversi.

La whitelist verifica che il dato abbia senso per l'applicazione.

Il prepared statement impedisce che il dato diventi codice SQL.

La bind variable tipata rafforza la distinzione tra valore e sintassi.

Esempio per uno user ID:

```java
if (!userid.matches("^[0-9]+$")) {
    throw new IllegalArgumentException("Invalid userid");
}

String query = "SELECT item, quantity, date, shipping FROM orders WHERE userid = ?";
PreparedStatement statement = connection.prepareStatement(query);
statement.setInt(1, Integer.parseInt(userid));
```

In questo esempio:
- la regex accetta solo cifre;
- il valore viene convertito a intero;
- la query usa un placeholder;
- `setInt` associa un intero;
- non c'è concatenazione di input nella query.

> ✅ Whitelist e prepared statement non sono alternative: si completano.

### **17. Altre Buone Pratiche Complementari**

La lezione si concentra su validazione e prepared statement.

Per completezza, queste tecniche vanno normalmente affiancate ad altre buone pratiche.

#### **17.1. Privilegio Minimo sul Database**

L'utente database usato dall'applicazione dovrebbe avere solo i permessi necessari.

Per esempio, se una pagina deve solo leggere ordini, non dovrebbe usare un account con permesso di:
- cancellare tabelle;
- creare utenti;
- modificare lo schema;
- eliminare database.

Questo non elimina la SQL injection, ma riduce il danno se una vulnerabilità esiste.

#### **17.2. Gestione Sicura degli Errori**

I messaggi di errore SQL non dovrebbero essere mostrati direttamente all'utente.

Messaggi troppo dettagliati possono rivelare:
- nome del DBMS;
- nomi di tabelle;
- nomi di colonne;
- struttura della query;
- stack trace;
- percorsi interni.

Queste informazioni aiutano l'attaccante a costruire payload più precisi.

#### **17.3. Centralizzazione dell'Accesso ai Dati**

È preferibile evitare che query SQL vengano costruite in modo sparso in molte parti dell'applicazione.

Centralizzare l'accesso ai dati rende più facile:
- applicare prepared statement ovunque;
- revisionare le query;
- imporre pattern sicuri;
- evitare duplicazioni vulnerabili.

#### **17.4. Test di Sicurezza**

Le applicazioni dovrebbero essere testate anche con input malevoli.

Esempi:

```sql
' OR 1=1 --
```

```sql
'; DROP TABLE users; --
```

```sql
NULL UNION ALL SELECT ...
```

Il test non deve limitarsi al caso "utente corretto", ma deve verificare anche input anomali.

> ⚠️ Queste pratiche sono complementari: non sostituiscono prepared statement e validazione server-side.

### **18. Caso Storico: Attacco del 2008 a Siti Microsoft ASP**

Nel transcript viene citato un caso storico avvenuto nel 2008.

Diversi web server Microsoft che usavano un particolare stile di pagine ASP vulnerabili furono colpiti da attacchi di SQL injection.

L'attacco seguiva una sequenza automatizzata.

#### **18.1. Ricerca dei Siti Vulnerabili**

In una prima fase veniva usato Google per individuare automaticamente siti che usavano uno stile di pagina ASP riconoscibile e vulnerabile.

L'idea era cercare pagine con pattern simili, probabilmente generate dallo stesso tipo di applicazione o dallo stesso stile di codice.

#### **18.2. Ricostruzione dei Campi Testuali**

In una seconda fase venivano eseguiti attacchi di SQL injection per recuperare i nomi dei campi del database che contenevano testo.

Questa fase serviva a capire dove poter iniettare contenuto persistente.

#### **18.3. Inserimento di Link Malevoli**

In una terza fase venivano effettuati attacchi di SQL injection per inserire, in ogni campo testuale trovato, un link verso un sito malevolo.

Nel transcript viene citato un sito cinese che ospitava codice JavaScript malevolo.

Il codice sfruttava vulnerabilità allora presenti in software diffusi, tra cui:
- Internet Explorer;
- RealPlayer;
- Instant Messenger usati comunemente in quel periodo.

#### **18.4. Effetto sugli Utenti**

Chi visitava le pagine compromesse o cliccava sul link malevolo poteva subire l'attacco lato client.

Questo caso mostra che una SQL injection può essere usata non solo per leggere o distruggere dati, ma anche per trasformare un sito vulnerabile in un veicolo di compromissione per altri utenti.

> ⚠️ Una SQL injection su un server può diventare il primo passo di un attacco più ampio contro i client che visitano il sito.

### **19. Impatti della SQL Injection**

La lezione ricorda che gli attacchi di SQL injection sono minacce gravi perché possono:
- rilasciare dati sensibili;
- alterare dati critici;
- danneggiare dati critici;
- dare all'attaccante accesso non autorizzato a un database;
- dare accesso non autorizzato ad aree protette dell'applicazione.

Questi impatti corrispondono alle proprietà fondamentali della sicurezza.

#### **19.1. Confidenzialità**

La confidenzialità viene violata quando l'attaccante legge dati che non dovrebbe vedere.

Esempi:
- password;
- dati personali;
- numeri di carta di credito;
- ordini;
- dati amministrativi.

#### **19.2. Integrità**

L'integrità viene violata quando l'attaccante modifica dati.

Esempi:
- modifica password;
- aggiunta di utenti;
- alterazione di prezzi;
- modifica di ordini;
- inserimento di link malevoli.

#### **19.3. Disponibilità**

La disponibilità viene violata quando l'attaccante rende il servizio inutilizzabile o distrugge dati.

Esempi:
- `DROP TABLE`;
- cancellazione di record;
- corruzione del database;
- blocco dell'applicazione.

> 📌 La SQL injection può colpire confidenzialità, integrità e disponibilità contemporaneamente.

### **20. Schema di Difesa Consigliato**

Una strategia coerente contro la SQL injection può essere riassunta così.

#### **20.1. Validare Tutto l'Input Lato Server**

Ogni dato ricevuto dal client deve essere controllato lato server.

Il controllo deve essere specifico per il campo.

Esempio:
- `userid`: intero positivo;
- `month`: valore da 1 a 12;
- `sort`: solo `asc` o `desc`;
- `category`: valore presente in un insieme definito;
- `username`: pattern ammesso dall'applicazione.

#### **20.2. Preferire Whitelist a Blacklist**

La whitelist definisce i valori ammessi.

La blacklist cerca di indovinare quelli vietati.

Per sicurezza, la prima è di norma più affidabile.

#### **20.3. Usare Prepared Statement Correttamente**

Ogni valore variabile deve essere associato tramite placeholder.

Forma corretta:

```sql
WHERE username = ?
```

Forma fragile:

```sql
WHERE username = '" + username + "'
```

#### **20.4. Usare Bind Variable Tipate**

I valori devono essere associati con metodi coerenti con il tipo.

Esempi:
- `setString`;
- `setInt`;
- `setDate`;
- `setBoolean`;
- `setBigDecimal`.

#### **20.5. Evitare Query Dinamiche Non Necessarie**

Ogni volta che la struttura della query viene costruita dinamicamente, aumenta il rischio di confondere dati e controllo.

Se una parte della query deve essere dinamica, per esempio un ordinamento, bisogna mapparla su valori ammessi.

Esempio:

```java
String orderBy;

if ("name".equals(inputOrder)) {
    orderBy = "name";
} else if ("date".equals(inputOrder)) {
    orderBy = "date";
} else {
    throw new IllegalArgumentException("Invalid order field");
}
```

In questo caso l'utente non fornisce direttamente SQL.

Fornisce una scelta che l'applicazione traduce in una colonna ammessa.

---

### **21. Errori Frequenti**

#### **21.1. Controllare Solo nel Browser**

Il controllo JavaScript lato client è facilmente aggirabile.

Serve per l'usabilità, non per garantire sicurezza.

#### **21.2. Filtrare Solo Alcuni Caratteri**

Rimuovere solo gli apici può non bastare.

Esistono altri metacaratteri, varianti sintattiche e contesti di query diversi.

#### **21.3. Usare PreparedStatement con Concatenazione**

Il nome dell'API non basta.

Se la query è costruita concatenando input, rimane vulnerabile.

#### **21.4. Fidarsi del Tipo HTML del Campo**

Un campo HTML di tipo `number` non garantisce che il server riceverà un numero.

L'attaccante può inviare una richiesta HTTP manuale con qualunque valore.

#### **21.5. Esporre Errori SQL**

Mostrare errori SQL completi aiuta l'attaccante a capire schema e DBMS.

Gli errori dettagliati vanno nei log, non nella risposta all'utente.

> ⚠️ Molti errori derivano dallo stesso equivoco: considerare affidabile ciò che arriva dal client.

### **22. Collegamento con le Lezioni Precedenti**

La lezione L1 ha introdotto SQL:
- query;
- `SELECT`;
- `FROM`;
- `WHERE`;
- comandi di modifica dei dati.

La lezione L2 ha mostrato come l'attaccante può sfruttare input non controllato:
- `OR 1=1`;
- commenti SQL;
- `DROP TABLE`;
- `UNION ALL SELECT`.

Questa lezione completa il quadro introducendo le contromisure:
- validazione dell'input;
- whitelist;
- prepared statement;
- bind variable tipate.

La relazione tra le tre lezioni è diretta:

> 📌 SQL injection nasce quando dati e codice SQL vengono mescolati; la difesa consiste nel validarli e separarli.

### **23. Sintesi Finale**

Le contromisure agli attacchi di SQL injection devono affrontare il problema alla radice.

Il problema non è solo la presenza di un apice o di una parola chiave pericolosa.

Il problema è che l'applicazione permette all'input dell'utente di diventare parte del comando SQL.

La difesa corretta richiede:
- validazione dell'input lato server;
- preferenza per whitelist rispetto a blacklist;
- espressioni regolari definite sui requisiti reali del campo;
- prepared statement;
- bind variable tipate;
- assenza di concatenazione tra input e query SQL;
- uso coerente di più misure.

Nel caso del blacklisting, bisogna ricordare che:
- può dimenticare casi pericolosi;
- può bloccare dati leciti;
- non dà garanzia completa.

Nel caso dei prepared statement, bisogna ricordare che:
- funzionano se la query è un template statico;
- i valori devono passare da placeholder;
- le bind variable devono essere usate davvero;
- concatenare stringhe dentro un `PreparedStatement` resta vulnerabile.

> ✅ L'idea chiave è mantenere distinti i dati dagli elementi di controllo della query SQL.

---

### **24. Recap per l'Esame**

> ✅ La SQL injection sfrutta input non controllato inserito direttamente in query SQL.

> ✅ I metacaratteri come apici, punto e virgola e commenti possono trasformare un dato in controllo SQL.

> ✅ Il blacklisting vieta elementi noti come pericolosi, ma è incompleto e può bloccare dati leciti.

> ✅ Il whitelisting accetta solo valori appartenenti a un insieme definito ed è più robusto.

> ✅ Le espressioni regolari permettono di descrivere pattern ammessi, ma devono essere precise rispetto al requisito.

> ✅ I prepared statement usano template SQL statici con placeholder.

> ✅ Le bind variable tipate associano i valori ai placeholder come dati, non come sintassi SQL.

> ✅ Usare `PreparedStatement` senza placeholder e continuando a concatenare stringhe non protegge.

> ✅ Una difesa efficace combina validazione lato server, whitelist e prepared statement con bind variable tipate.

> ⚠️ Il controllo lato client non basta: ogni input rilevante deve essere validato e trattato in sicurezza lato server.
