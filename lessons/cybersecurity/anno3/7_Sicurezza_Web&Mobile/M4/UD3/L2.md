# L2 — SQL Injection: Funzionamento dell'Attacco

> 📌 La **SQL injection** è un attacco in cui l'input fornito dall'utente viene inserito senza controllo dentro una query SQL lato server. L'attaccante sfrutta questa situazione per cambiare la semantica della query, ottenendo dati, aggirando l'autenticazione o modificando il database.

---

### **1. Obiettivo della Lezione**

In questa lezione viene descritto il funzionamento di un attacco di tipo **SQL injection**.

Nella lezione precedente sono stati introdotti gli elementi minimi di SQL necessari per capire l'attacco:
- tabelle relazionali;
- attributi e record;
- query `SELECT`;
- clausole `FROM` e `WHERE`;
- operatori booleani;
- comandi di modifica o distruzione dei dati, come `INSERT`, `UPDATE` e `DROP`.

Questa lezione usa quegli elementi per mostrare come un'applicazione web vulnerabile possa trasformare un normale dato di input in una parte del comando SQL eseguito dal server.

L'idea fondamentale è questa:

> 📌 l'attaccante non attacca direttamente il database; attacca l'applicazione web che costruisce male le query verso il database.

---

### **2. Contesto dell'Attacco**

Molti siti web dinamici costruiscono le pagine richieste dal client recuperando contenuti da una base di dati.

Il caso tipico è:
- il client invia una richiesta HTTP;
- l'applicazione lato server riceve la richiesta;
- uno script lato server interroga un database;
- il database restituisce dati;
- il server costruisce dinamicamente la pagina HTML da inviare al client.

Esempi di dati recuperati da un database:
- credenziali utente;
- profili personali;
- ordini di un cliente;
- carrelli della spesa;
- prodotti disponibili;
- pagamenti;
- dati amministrativi.

La SQL injection sfrutta il punto in cui l'applicazione server-side passa da:

> dato ricevuto dall'utente

a:

> frammento della query SQL eseguita sul database.

Se questa trasformazione avviene senza controlli, il dato dell'utente può diventare codice SQL.

---

### **3. Idea di Base della SQL Injection**

L'obiettivo dell'attacco è **modificare la semantica della query SQL lato server**.

In condizioni normali, il programmatore immagina una query con un significato preciso.

Per esempio:

```sql
SELECT *
FROM users
WHERE username = 'alice' AND password = 'passwordInserita';
```

Il significato previsto è:

> cerca un utente che abbia esattamente quello username e quella password.

In una SQL injection, l'attaccante inserisce caratteri e parole chiave SQL dentro i campi di input, in modo che la query finale non significhi più ciò che il programmatore aveva previsto.

Il problema nasce quando l'applicazione costruisce la query tramite semplice concatenazione di stringhe:

```php
$query = "SELECT * FROM users WHERE username = '" . $username . "' AND password = '" . $password . "'";
```

In questo caso, se `$username` e `$password` arrivano direttamente dall'utente, l'utente controlla una porzione della query.

> ⚠️ La vulnerabilità non è SQL in sé. La vulnerabilità è l'inserimento diretto di input non fidato dentro un comando SQL.

---

### **4. Input Non Fidato**

L'input fornito dal client deve essere considerato **non fidato**.

Un'applicazione web può ricevere input da molti punti:
- campi di un form HTML;
- parametri nella query string dell'URL;
- parametri inviati con metodo `POST`;
- cookie;
- header HTTP;
- valori nascosti nei form;
- dati inviati da JavaScript.

Nel transcript viene evidenziato soprattutto il caso del form e della query dell'URL:
- nel form l'utente può inserire username, password o altri dati;
- nella query dell'URL l'utente può manipolare parametri visibili quando viene usato il metodo `GET`.

Esempio di URL con parametro:

```text
https://example.com/list_orders.php?userid=42
```

Il valore `42` sembra un semplice identificativo utente.

Però, se lo script lato server lo inserisce direttamente in SQL, l'attaccante può sostituirlo con un frammento SQL.

> 📌 La SQL injection è possibile quando il server tratta l'input del client come se fosse già sicuro e semanticamente valido.

---

### **5. Flusso Generale dell'Attacco**

Il funzionamento di base di una SQL injection può essere riassunto in quattro fasi.

#### **5.1. Inserimento dell'Input Malevolo**

L'attaccante inserisce input malevolo:
- in un campo di un form;
- oppure nella parte di query dell'URL, quando l'applicazione usa parametri `GET`;
- oppure in qualunque altro punto usato dal server per costruire query SQL.

L'attaccante può essere:
- il client malevolo che interagisce direttamente con il sito;
- oppure un soggetto che riesce a intromettersi nella comunicazione e alterare la richiesta.

#### **5.2. Costruzione della Query Vulnerabile**

Il server riceve la richiesta e recupera i parametri inseriti dall'utente.

Se l'applicazione è vulnerabile, inserisce questi parametri direttamente dentro una stringa SQL.

Per esempio:

```php
$query = "SELECT * FROM users WHERE username = '$username' AND password = '$password'";
```

Se `$username` contiene solo un nome utente normale, la query rimane quella prevista.

Se invece `$username` contiene anche sintassi SQL, la query viene trasformata.

#### **5.3. Esecuzione sul Database**

Il server invia la query al database.

Il database non sa quali parti della query provengano dal programmatore e quali parti provengano dall'utente.

Il database vede solo un comando SQL completo.

Quindi lo interpreta ed esegue secondo le regole del linguaggio SQL.

#### **5.4. Effetto dell'Attacco**

Poiché la semantica della query è stata alterata, possono verificarsi effetti come:
- rilascio di dati sensibili al client;
- bypass dell'autenticazione;
- modifica di dati esistenti;
- inserimento di nuovi dati;
- cancellazione di tabelle;
- distruzione parziale o totale della base di dati;
- combinazione di più effetti nello stesso attacco.

> ⚠️ Dal punto di vista del database, l'attacco non è necessariamente distinguibile da una normale query SQL: il problema è che la query non avrebbe dovuto essere costruita in quel modo.

---

### **6. Condizioni Necessarie per l'Attacco**

Perché una SQL injection possa essere effettuata devono essere presenti alcune condizioni.

#### **6.1. Presenza di Input Controllabile dal Client**

Il client deve poter inserire dati.

Il caso più comune è una pagina di login in cui l'utente inserisce:
- username;
- password.

Il form HTML invia questi dati al server.

In generale, qualunque campo di input può diventare un vettore di SQL injection se viene usato per costruire query SQL.

#### **6.2. Script Lato Server Vulnerabile**

Lato server deve esserci uno script che:
- legge i dati ricevuti dal form;
- li usa per costruire una query SQL;
- non li valida;
- non li filtra;
- non li separa strutturalmente dal codice SQL.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

Un esempio concettuale di codice vulnerabile è:

```php
$username = $_POST["username"];
$password = $_POST["password"];

$query = "SELECT * FROM users
          WHERE username = '$username'
          AND password = '$password'";
```

Il problema è che `username` e `password` vengono inseriti direttamente nel comando SQL.

> ⚠️ La semplice concatenazione di stringhe tra query SQL e input utente è il punto critico dell'attacco.

#### **6.3. Presenza di un Database Lato Server**

Deve esserci un database interrogato dallo script.

Nel caso della pagina di login, è sufficiente che esista una tabella come:
- `users`;
- con almeno un campo `username`;
- e almeno un campo `password`.

L'attaccante non ha bisogno di accedere direttamente al database.

Gli basta poter influenzare la query che l'applicazione invia al database.

#### **6.4. Mancato Controllo dell'Input**

La condizione decisiva è l'assenza di controlli.

L'attacco diventa possibile quando il server:
- non verifica la forma dell'input;
- non impone vincoli sul tipo di dato;
- non neutralizza caratteri speciali;
- non usa query parametrizzate;
- non separa dati e codice SQL.

> 📌 La SQL injection è una vulnerabilità di confine: nasce nel punto in cui dati esterni entrano in una query interna.

### **7. Primo Esempio: Login Vulnerabile**

Consideriamo una pagina di login.

L'utente inserisce:
- username;
- password.

L'applicazione deve controllare se esiste una riga nella tabella `users` con quei valori.

La query prevista dal programmatore è:

```sql
SELECT *
FROM users
WHERE username = 'valoreUsername'
AND password = 'valorePassword';
```

Questa query ritorna:
- una riga, se le credenziali sono corrette;
- una tabella vuota, se username o password sono errati.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **7.1. Funzionamento Normale**

Il funzionamento normale può essere descritto così.

1. L'utente inserisce username e password nella finestra di login.
2. Il browser invia una richiesta HTTP con i due parametri.
3. Il web server costruisce la query SQL usando quei due parametri.
4. Il web server interroga il database.
5. Il database restituisce una riga oppure una tabella vuota.
6. Se la tabella è vuota, il server nega l'accesso.
7. Se la tabella contiene una riga valida, il server concede l'accesso alla risorsa richiesta.

Nel caso di fallimento, il server può inviare una risposta HTTP con codice:

```text
403 Forbidden
```

Nel caso di successo, il server restituisce la pagina HTML richiesta.

#### **7.2. Query Normale**

Supponiamo che l'utente inserisca:
- username: `alice`;
- password: `segreta`.

La query prodotta sarà:

```sql
SELECT *
FROM users
WHERE username = 'alice'
AND password = 'segreta';
```

Il significato è corretto:

> accedi solo se esiste un utente `alice` con password `segreta`.

---

### **8. Bypass dell'Autenticazione con Tautologia**

Ora supponiamo che l'attaccante inserisca nel campo username un valore costruito per alterare la clausola `WHERE`.

Un esempio tipico è:

```sql
' OR 1=1 --
```

La password può essere qualunque valore, perché l'attaccante cercherà di commentare la parte della query che la controlla.

La query finale può diventare:

```sql
SELECT *
FROM users
WHERE username = ''
OR 1=1 --'
AND password = 'qualunque';
```

Il frammento importante è:

```sql
OR 1=1
```

La condizione `1=1` è sempre vera.

Quindi la clausola `WHERE` diventa vera per tutte le righe.

Il frammento:

```sql
--
```

inizia un commento SQL.

Di conseguenza, ciò che segue viene ignorato dal database, compreso il controllo sulla password.

> 📌 La tautologia `1=1` rende vera la condizione; il commento elimina la parte residua della query che ostacolerebbe l'attacco.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **8.1. Effetto sul Risultato**

La query non ritorna più solo la riga dell'utente con username e password corretti.

Può invece ritornare:
- l'intera tabella degli utenti;
- oppure comunque almeno una riga sufficiente a far credere all'applicazione che il login sia riuscito.

Se l'applicazione controlla soltanto che il risultato non sia vuoto, l'attaccante viene autenticato.

Questo può accadere anche se:
- l'utente non esiste nel database;
- la password è sbagliata;
- il campo password è vuoto;
- l'attaccante non conosce alcuna credenziale valida.

#### **8.2. Perché il Bypass Funziona**

Il bypass funziona perché l'applicazione ragiona così:

> se la query restituisce almeno una riga, allora l'utente è autenticato.

Ma l'attaccante ha cambiato il significato della query da:

> trova l'utente con questo username e questa password

a:

> trova righe per cui la condizione è vera, e la condizione è sempre vera.

> ⚠️ L'errore non è solo nel database. È anche nella logica applicativa che considera il risultato della query come prova sufficiente di autenticazione.

---

### **9. Ruolo degli Apici e dei Commenti**

Gli apici sono importanti perché molte query confrontano stringhe racchiudendole tra apici.

Esempio:

```sql
WHERE username = 'alice'
```

Se l'applicazione concatena l'input dentro gli apici, l'attaccante può provare a chiudere la stringa prima del previsto.

Per esempio:

```sql
' OR 1=1 --
```

Il primo apice chiude la stringa.

Poi viene inserita una condizione SQL:

```sql
OR 1=1
```

Infine il commento elimina il resto della query.

In molti dialetti SQL, `--` introduce un commento fino alla fine della riga.

> ⚠️ La sintassi precisa dei commenti può dipendere dal DBMS. In alcuni casi dopo `--` è richiesto uno spazio. Il principio resta lo stesso: l'attaccante cerca di neutralizzare la parte residua della query.

---

### **10. Secondo Esempio: Cancellazione della Tabella Utenti**

Una SQL injection non serve solo a leggere dati o bypassare un login.

Può anche modificare o distruggere dati.

Supponiamo che l'attaccante inserisca un input che chiude la query corrente e aggiunge un comando distruttivo:

```sql
'; DROP TABLE users; --
```

La query risultante può assumere una forma di questo tipo:

```sql
SELECT *
FROM users
WHERE username = '';
DROP TABLE users; --'
AND password = 'qualunque';
```

L'effetto previsto è:
- la prima query può non autenticare l'utente;
- però viene eseguito anche `DROP TABLE users`;
- la tabella degli utenti viene rimossa dal database.

Nel transcript viene evidenziato che l'accesso può anche essere negato, ma con un effetto collaterale molto grave:

> la tabella relativa agli utenti è stata cancellata.

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **10.1. Dipendenza dal DBMS e dal Driver**

Questo esempio presuppone che l'ambiente consenta l'esecuzione di più istruzioni SQL nella stessa richiesta.

Non tutti i driver, framework o DBMS lo permettono nello stesso modo.

Il punto didattico resta però fondamentale:

> se l'input può diventare codice SQL, l'attaccante può tentare non solo di leggere dati, ma anche di eseguire comandi che modificano lo stato del database.

#### **10.2. Altri Comandi Pericolosi**

Con lo stesso principio, un attaccante potrebbe tentare di usare:
- `INSERT INTO`, per aggiungere nuovi record;
- `UPDATE`, per modificare dati esistenti;
- `DELETE`, per eliminare record;
- `DROP TABLE`, per eliminare una tabella;
- `DROP DATABASE`, per eliminare un intero database, se i permessi lo consentono.

Esempi di obiettivi:
- aggiungere un nuovo utente;
- modificare una password;
- assegnarsi privilegi amministrativi;
- cancellare ordini;
- alterare prezzi;
- distruggere tabelle.

> ⚠️ L'impatto effettivo dipende anche dai permessi dell'utente database usato dall'applicazione. Se l'applicazione si connette con privilegi eccessivi, il danno potenziale aumenta.

---

### **11. Necessità di Conoscere la Struttura del Database**

Nel transcript viene sottolineato un limite pratico di molti attacchi di SQL injection:

> l'attaccante deve conoscere, o riuscire a ricostruire, la struttura del database.

Per costruire un attacco efficace può servire conoscere:
- nomi delle tabelle;
- nomi degli attributi;
- numero di colonne restituite da una query;
- tipi delle colonne;
- relazioni tra tabelle;
- DBMS usato;
- comportamento dei commenti;
- permessi dell'utente database.

Per esempio, per eseguire:

```sql
DROP TABLE users;
```

l'attaccante deve sapere o indovinare che esiste una tabella chiamata `users`.

Per eseguire:

```sql
SELECT name, creditNumber, expMonth, expYear
FROM creditCards;
```

deve sapere o indovinare che esiste una tabella `creditCards` con quegli attributi.

> 💡 Anche quando l'attaccante non conosce inizialmente lo schema, errori applicativi, messaggi diagnostici e comportamenti diversi del server possono aiutarlo a dedurlo. Per questo i messaggi di errore dettagliati non dovrebbero essere esposti agli utenti finali.

### **12. Terzo Esempio: Estrazione di Dati con UNION**

L'ultimo esempio del transcript riguarda uno scenario diverso dal login.

Supponiamo di avere un sito di e-commerce che permette a un utente di visualizzare la lista dei propri ordini.

La pagina web richiama uno script, per esempio:

```text
list_orders
```

passando come parametro lo user ID dell'utente.

Un URL possibile è:

```text
https://example.com/list_orders?userid=42
```

Lo script lato server usa il valore di `userid` per costruire una query.

Esempio concettuale:

```sql
SELECT item, quantity, date, shipping
FROM orders
WHERE userid = 42;
```

Questa query dovrebbe restituire gli ordini dell'utente `42`.

#### **12.1. Script Vulnerabile**

La vulnerabilità nasce se lo script costruisce la query concatenando direttamente il parametro:

```php
$userid = $_GET["userid"];

$query = "SELECT item, quantity, date, shipping
          FROM orders
          WHERE userid = " . $userid;
```

Se `userid` è davvero un numero, la query funziona come previsto.

Se però l'attaccante inserisce SQL al posto del numero, il significato della query cambia.

### **13. Operatore UNION**

L'attacco descritto usa l'operatore SQL `UNION`.

`UNION` combina i risultati di due query `SELECT`.

In forma astratta:

```sql
SELECT colonne
FROM tabella1
UNION
SELECT colonne
FROM tabella2;
```

Perché l'operazione sia valida, le due `SELECT` devono essere compatibili.

In particolare:
- devono restituire lo stesso numero di colonne;
- le colonne corrispondenti devono avere tipi compatibili;
- il risultato finale viene trattato come un'unica tabella.

Nel transcript viene usato `UNION ALL`.

La differenza principale è:
- `UNION` tende a eliminare duplicati;
- `UNION ALL` conserva anche eventuali duplicati.

Per l'attacco, l'aspetto importante è che i dati della seconda query vengono inseriti nel risultato visualizzato dalla pagina.

> 📌 Con `UNION`, l'attaccante cerca di far comparire nella pagina dati provenienti da una tabella diversa da quella prevista dal programmatore.

---

### **14. Attacco UNION su Lista Ordini**

La query prevista dal programmatore è:

```sql
SELECT item, quantity, date, shipping
FROM orders
WHERE userid = valoreInserito;
```

L'attaccante inserisce come valore di `userid` un frammento come:

```sql
NULL UNION ALL SELECT name, creditNumber, expMonth, expYear FROM creditCards
```

La query finale diventa:

```sql
SELECT item, quantity, date, shipping
FROM orders
WHERE userid = NULL
UNION ALL
SELECT name, creditNumber, expMonth, expYear
FROM creditCards;
```

<!-- INSERT INSTRUCTOR SLIDE/DIAGRAM HERE -->

#### **14.1. Prima SELECT**

La prima `SELECT` è:

```sql
SELECT item, quantity, date, shipping
FROM orders
WHERE userid = NULL
```

Questa parte non restituisce gli ordini dell'utente.

Nel transcript viene descritta come una query che non ritorna nulla.

Il punto è che l'attaccante non è interessato agli ordini.

Vuole usare la struttura della query originale per agganciare una seconda `SELECT`.

#### **14.2. Seconda SELECT**

La seconda `SELECT` è:

```sql
SELECT name, creditNumber, expMonth, expYear
FROM creditCards;
```

Questa query legge dati dalla tabella delle carte di credito.

Le colonne scelte sono quattro:
- nome dell'utente;
- numero della carta di credito;
- mese di scadenza;
- anno di scadenza.

La query originale restituiva anch'essa quattro colonne:
- `item`;
- `quantity`;
- `date`;
- `shipping`.

Il numero di colonne coincide.

Quindi l'operatore `UNION ALL` può combinare i risultati.

#### **14.3. Risultato Visibile all'Attaccante**

Il server costruisce la pagina come se stesse visualizzando una lista di ordini.

Però il risultato ricevuto dal database contiene anche, o soprattutto, dati della tabella `creditCards`.

Di conseguenza, la pagina restituita al client malevolo può contenere:
- nomi degli utenti;
- numeri di carta di credito;
- date di scadenza;
- altre informazioni sensibili.

> ⚠️ Questo esempio mostra una violazione della confidenzialità: l'attaccante riesce a leggere dati che non avrebbe dovuto vedere.

---

### **15. Perché l'Attacco UNION è Potente**

L'attacco con `UNION` è particolarmente importante perché non si limita ad alterare una condizione booleana.

Permette di cambiare la sorgente dei dati restituiti.

La pagina si aspetta una tabella con un certo numero di colonne.

L'attaccante prova a costruire una seconda tabella compatibile, ma contenente dati diversi.

Se l'applicazione visualizza automaticamente il risultato della query, quei dati vengono mostrati nel browser.

Il meccanismo sfrutta tre fatti:
- il risultato di una query SQL è una tabella;
- `UNION` combina tabelle con struttura compatibile;
- l'applicazione si fida del risultato restituito dal database.

> 💡 L'attacco funziona perché l'applicazione non distingue semanticamente tra "ordini dell'utente" e "righe qualsiasi restituite dalla query": entrambe arrivano come risultato SQL e vengono usate per costruire HTML.

### **16. Tipologie di Impatto**

Gli esempi della lezione mostrano tre grandi categorie di impatto.

#### **16.1. Bypass dell'Autenticazione**

Nel caso del login vulnerabile, l'attaccante usa:
- apici;
- `OR`;
- tautologia;
- commento.

L'effetto è superare il controllo username-password.

#### **16.2. Distruzione o Modifica dei Dati**

Nel caso del comando `DROP TABLE`, l'attaccante cerca di eseguire un comando aggiuntivo.

L'effetto può essere:
- cancellazione di tabelle;
- modifica di dati;
- inserimento di nuovi utenti;
- alterazione di password;
- compromissione dell'integrità del database.

#### **16.3. Esfiltrazione di Dati**

Nel caso dell'attacco con `UNION`, l'attaccante legge dati da una tabella diversa da quella prevista.

L'effetto è:
- divulgazione di dati personali;
- furto di dati di pagamento;
- violazione della privacy;
- compromissione della confidenzialità.

> 📌 La SQL injection può compromettere tutte e tre le proprietà fondamentali della sicurezza: confidenzialità, integrità e disponibilità.

---

### **17. Il Ruolo del Server Vulnerabile**

È importante distinguere tra:
- ciò che fa il client;
- ciò che fa il server;
- ciò che fa il database.

Il client malevolo invia input costruito appositamente.

Il server vulnerabile commette l'errore decisivo:
- prende l'input;
- lo inserisce direttamente nella query;
- invia la query al database.

Il database esegue la query che riceve.

Quindi la vulnerabilità principale si trova nell'applicazione lato server.

> ⚠️ Non bisogna pensare alla SQL injection come a un "bug del database". È una vulnerabilità dell'applicazione che usa il database in modo insicuro.

---

### **18. Dato e Codice nella Stessa Stringa**

Il cuore concettuale della SQL injection è la confusione tra **dato** e **codice**.

Il programmatore vorrebbe che l'input dell'utente fosse solo un valore.

Per esempio:

```text
alice
```

oppure:

```text
42
```

Ma, se la query è costruita come stringa unica, l'input può essere interpretato come sintassi SQL.

Per esempio:

```sql
' OR 1=1 --
```

oppure:

```sql
NULL UNION ALL SELECT ...
```

In quel momento l'input smette di essere un semplice dato e diventa parte del programma SQL.

> 📌 La difesa corretta dovrà impedire che i dati dell'utente possano cambiare la struttura sintattica della query.

---

### **19. Perché la Vulnerabilità è Frequente**

Nel transcript viene osservato che il caso in cui il server non convalida i dati in input è ancora molto frequente.

Le cause tipiche sono:
- sviluppo rapido di applicazioni web;
- uso ingenuo della concatenazione di stringhe;
- fiducia eccessiva nei form HTML;
- mancata validazione lato server;
- assenza di query parametrizzate;
- gestione errata degli errori SQL;
- privilegi eccessivi assegnati all'utente database usato dall'applicazione;
- scarsa separazione tra logica applicativa e accesso ai dati.

Un errore comune è pensare che il controllo lato client sia sufficiente.

Per esempio, un form HTML può indicare che un campo deve contenere un numero.

Ma l'attaccante non è obbligato a usare il browser nel modo previsto.

Può costruire direttamente una richiesta HTTP modificata.

> ⚠️ Ogni controllo di sicurezza rilevante deve essere fatto lato server. Il client non è un confine di fiducia.

---

### **20. Collegamento con HTTP e Applicazioni Web**

La SQL injection si inserisce nella catena studiata nelle lezioni precedenti:

1. il browser invia una richiesta HTTP;
2. la richiesta contiene parametri controllabili dall'utente;
3. il server elabora la richiesta con uno script;
4. lo script interroga un database;
5. il risultato viene trasformato in una risposta HTTP.

Il punto vulnerabile è tra lo script e il database.

Tuttavia l'attacco parte dal livello web:
- un campo di login;
- un URL;
- una richiesta manipolata;
- un parametro apparentemente innocuo.

Per questo la SQL injection è una vulnerabilità tipica delle applicazioni web dinamiche.

> 💡 Il protocollo HTTP trasporta l'input; lo script lato server lo trasforma in SQL; il database esegue la query risultante.

---

### **21. Schema Logico dell'Attacco**

Una SQL injection può essere letta come una trasformazione.

Query prevista:

```sql
SELECT *
FROM users
WHERE username = '<dato>'
AND password = '<dato>';
```

Input malevolo:

```sql
' OR 1=1 --
```

Query ottenuta:

```sql
SELECT *
FROM users
WHERE username = ''
OR 1=1 --'
AND password = '<dato>';
```

Semantica prevista:

> controlla username e password.

Semantica ottenuta:

> restituisci righe perché la condizione contiene una tautologia.

Questa trasformazione è il punto centrale dell'attacco.

---

### **22. Errori Concettuali da Evitare**

#### **22.1. Pensare che Bastino Form Ben Fatti**

Un form può avere campi, vincoli e controlli JavaScript.

Ma l'attaccante può inviare richieste HTTP manuali o modificate.

Quindi il form non protegge il server.

#### **22.2. Pensare che il Database Capisca l'Intenzione del Programmatore**

Il database non conosce l'intenzione del programmatore.

Esegue la query che riceve.

Se la query contiene `OR 1=1`, il database applica `OR 1=1`.

#### **22.3. Pensare che il Problema Riguardi Solo le Password**

La SQL injection non riguarda solo login e password.

Può colpire:
- ricerca prodotti;
- filtri per categoria;
- pagine di dettaglio;
- liste ordini;
- pannelli amministrativi;
- API;
- campi nascosti;
- parametri nei link.

#### **22.4. Pensare che Serva Sempre un Accesso Riuscito**

Nel caso `DROP TABLE`, l'accesso può anche fallire.

L'attacco può comunque produrre un effetto collaterale dannoso sul database.

> ⚠️ Una risposta "accesso negato" non implica necessariamente che il database non sia stato alterato.

---

### **23. Lessico Essenziale**

#### **23.1. Query Vulnerabile**

Una query è vulnerabile quando incorpora input non fidato senza separarlo dal codice SQL.

#### **23.2. Payload**

Il payload è l'input malevolo inserito dall'attaccante.

Esempi:

```sql
' OR 1=1 --
```

```sql
'; DROP TABLE users; --
```

```sql
NULL UNION ALL SELECT name, creditNumber, expMonth, expYear FROM creditCards
```

#### **23.3. Tautologia**

Una tautologia è una condizione sempre vera.

Esempio:

```sql
1=1
```

#### **23.4. Commento SQL**

Un commento SQL consente di ignorare la parte successiva della query.

Esempio:

```sql
--
```

#### **23.5. Query Stacking**

Il query stacking consiste nel tentativo di eseguire più istruzioni SQL nella stessa richiesta.

Esempio:

```sql
SELECT ...;
DROP TABLE users;
```

#### **23.6. UNION-Based Injection**

Una union-based injection usa `UNION` o `UNION ALL` per aggiungere al risultato della query dati provenienti da un'altra tabella.

---

### **24. Sintesi Finale**

Una SQL injection si verifica quando:
- l'applicazione riceve input dall'utente;
- l'input non viene controllato;
- l'input viene concatenato dentro una query SQL;
- il database esegue la query risultante;
- la semantica della query viene alterata dall'attaccante.

Gli esempi principali della lezione sono:
- bypass del login con `OR 1=1`;
- eliminazione della tabella utenti con `DROP TABLE users`;
- estrazione di dati sensibili con `UNION ALL SELECT`.

La causa comune è sempre la stessa:

> 📌 input non fidato inserito direttamente in una query lato server.

Dal punto di vista della sicurezza, l'attacco può produrre:
- accesso non autorizzato;
- violazione della confidenzialità;
- modifica o distruzione dei dati;
- compromissione della disponibilità del servizio;
- esposizione di informazioni personali e finanziarie.

---

### **25. Recap per l'Esame**

> ✅ Una SQL injection modifica la semantica della query SQL costruita dal server.

> ✅ L'attacco è possibile quando input non fidato viene concatenato direttamente nella query.

> ✅ Nel login vulnerabile, `OR 1=1` rende vera la condizione e `--` può commentare il controllo sulla password.

> ✅ Con `DROP TABLE` l'attaccante può tentare di distruggere dati, se DBMS, driver e permessi lo permettono.

> ✅ Con `UNION ALL SELECT` l'attaccante può far comparire nel risultato dati provenienti da tabelle diverse.

> ✅ Il database esegue la query ricevuta; la vulnerabilità sta nella costruzione insicura della query da parte dell'applicazione web.

> ⚠️ Non basta controllare l'input lato client: il controllo di sicurezza deve avvenire lato server.

> ⚠️ Una query SQL costruita per concatenazione di stringhe con parametri utente è un forte indicatore di vulnerabilità.
