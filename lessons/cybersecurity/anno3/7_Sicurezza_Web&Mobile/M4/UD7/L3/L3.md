# L3 — Uso di GPG: Esercizi su Cifratura, Chiavi e Firma

> 📌 **GnuPG** (*GNU Privacy Guard*, o **GPG**) è un'implementazione libera dello standard **OpenPGP**. Permette di cifrare, decifrare, firmare e verificare file o messaggi usando gli stessi principi studiati nella lezione su PGP: chiavi simmetriche di sessione, chiavi pubbliche/private, keyring, fingerprint e web of trust.

---

### **1. Obiettivo della Lezione**

Questa lezione raccoglie le tracce degli esercizi su GPG e ne fornisce una soluzione guidata e completa.

Gli esercizi servono a mettere in pratica i concetti studiati nella lezione precedente su PGP:
- cifratura simmetrica;
- cifratura asimmetrica;
- generazione di coppie di chiavi;
- esportazione e importazione di chiavi pubbliche;
- verifica della fingerprint;
- firma di chiavi pubbliche;
- firma e cifratura di un messaggio;
- decifratura e verifica della firma.

GPG è già installato nella versione di Ubuntu suggerita per la macchina virtuale di laboratorio.

Il manuale indicato nella traccia è:

```text
https://www.gnupg.org/gph/it/
```

> 💡 PGP è il modello concettuale; GPG è lo strumento pratico con cui eseguire cifratura, firma e gestione delle chiavi.

### **2. Richiamo Teorico Essenziale**

Prima degli esercizi è utile richiamare alcuni punti della lezione precedente.

PGP/GPG può operare in tre modi:
- solo cifratura;
- solo firma;
- firma e cifratura.

La **cifratura simmetrica** usa una sola chiave o passphrase condivisa.

Chi cifra e chi decifra devono conoscere lo stesso segreto.

La **cifratura asimmetrica** usa una coppia di chiavi:
- chiave pubblica;
- chiave privata.

Per inviare un messaggio confidenziale a Bob, Alice cifra usando la **chiave pubblica di Bob**.

Solo Bob può decifrare, usando la propria **chiave privata**.

La **firma digitale** funziona nella direzione opposta:
- il mittente firma usando la propria chiave privata;
- il destinatario verifica usando la chiave pubblica del mittente.

La firma garantisce:
- integrità del messaggio;
- autenticità rispetto alla chiave pubblica usata per verificare;
- non ripudio, nel limite in cui la chiave privata sia rimasta sotto controllo del proprietario.

> ⚠️ La verifica della firma è significativa solo se la chiave pubblica del mittente è autentica. Per questo negli esercizi si controlla la fingerprint e si firma la chiave importata.

### **4. Esercizio 1 — Traccia**

Creare un file di testo `messaggio.txt` con un qualsiasi contenuto.

Usando GPG:
- cifrare il file con crittografia simmetrica usando algoritmi diversi, cioè **AES**, **3DES** e **Blowfish**, generando file diversi a seconda dell'algoritmo usato;
- decifrare i file generati al punto precedente.

---

### **5. Esercizio 1 — Soluzione: Creazione del File**

Creiamo un file di testo di prova:

```bash
cat > messaggio.txt <<'EOF'
Questo è un messaggio di prova per l'esercizio su GPG.
Il file verrà cifrato con algoritmi simmetrici diversi.
EOF
```

Verifichiamo il contenuto:

```bash
cat messaggio.txt
```

In questa fase il file è in chiaro.

Chiunque abbia accesso al file può leggerne il contenuto.

---

### **6. Esercizio 1 — Soluzione: Cifratura Simmetrica con AES**

Per cifrare il file con algoritmo simmetrico **AES**:

```bash
gpg --symmetric --cipher-algo AES --output messaggio-aes.gpg messaggio.txt
```

Significato delle opzioni:
- `--symmetric` indica che si usa cifratura simmetrica;
- `--cipher-algo AES` seleziona l'algoritmo AES;
- `--output messaggio-aes.gpg` indica il file cifrato da produrre;
- `messaggio.txt` è il file in chiaro da cifrare.

GPG chiede una passphrase.

La stessa passphrase dovrà essere usata per decifrare.

Il file prodotto:

```text
messaggio-aes.gpg
```

contiene il messaggio cifrato.

> 📌 In modalità simmetrica non serve alcuna chiave pubblica: la sicurezza dipende dalla passphrase scelta.

---

### **7. Esercizio 1 — Soluzione: Cifratura Simmetrica con 3DES**

Per cifrare con **3DES**:

```bash
gpg --symmetric --cipher-algo 3DES --output messaggio-3des.gpg messaggio.txt
```

Il file prodotto è:

```text
messaggio-3des.gpg
```

Il procedimento è lo stesso usato per AES.

Cambia l'algoritmo simmetrico.

3DES è storicamente importante perché nasce come rafforzamento di DES, ma oggi non è la scelta preferibile per nuovi sistemi.

Nell'esercizio serve a osservare come GPG permetta di selezionare algoritmi diversi.

---

### **8. Esercizio 1 — Soluzione: Cifratura Simmetrica con Blowfish**

Per cifrare con **Blowfish**:

```bash
gpg --symmetric --cipher-algo BLOWFISH --output messaggio-blowfish.gpg messaggio.txt
```

Il file prodotto è:

```text
messaggio-blowfish.gpg
```

Anche qui il modello è simmetrico:
- una passphrase protegge il contenuto;
- la stessa passphrase serve per decifrare;
- non intervengono chiavi pubbliche o private.

---

### **9. Esercizio 1 — Soluzione: Controllo dei File Prodotti**

Dopo le tre cifrature, si possono elencare i file:

```bash
ls -l
```

Ci si aspetta di vedere almeno:

```text
messaggio.txt
messaggio-aes.gpg
messaggio-3des.gpg
messaggio-blowfish.gpg
```

I file `.gpg` sono file cifrati.

Per ispezionarli superficialmente:

```bash
file messaggio-aes.gpg
file messaggio-3des.gpg
file messaggio-blowfish.gpg
```

Non devono essere leggibili come testo normale.

Se si prova:

```bash
cat messaggio-aes.gpg
```

si ottiene contenuto binario o non interpretabile come messaggio in chiaro.

### **10. Esercizio 1 — Soluzione: Decifratura dei File**

Per decifrare il file cifrato con AES:

```bash
gpg --output messaggio-aes-decifrato.txt --decrypt messaggio-aes.gpg
```

Per decifrare il file cifrato con 3DES:

```bash
gpg --output messaggio-3des-decifrato.txt --decrypt messaggio-3des.gpg
```

Per decifrare il file cifrato con Blowfish:

```bash
gpg --output messaggio-blowfish-decifrato.txt --decrypt messaggio-blowfish.gpg
```

GPG chiede la passphrase usata in fase di cifratura.

Se la passphrase è corretta, vengono prodotti tre file in chiaro:

```text
messaggio-aes-decifrato.txt
messaggio-3des-decifrato.txt
messaggio-blowfish-decifrato.txt
```

### **11. Esercizio 1 — Soluzione: Verifica della Correttezza**

Per verificare che la decifratura abbia recuperato esattamente il messaggio originale:

```bash
diff -u messaggio.txt messaggio-aes-decifrato.txt
diff -u messaggio.txt messaggio-3des-decifrato.txt
diff -u messaggio.txt messaggio-blowfish-decifrato.txt
```

Se `diff` non stampa nulla, i file sono identici.

Questo conferma che:
- la cifratura ha protetto il contenuto;
- la passphrase corretta permette di decifrare;
- il messaggio originale viene recuperato.

> ✅ L'esercizio 1 dimostra la modalità simmetrica di GPG: un file viene cifrato con una passphrase e può essere decifrato solo da chi conosce quella passphrase.

### **12. Esercizio 1 — Interpretazione Teorica**

Questo esercizio usa solo cifratura simmetrica.

Non vengono create coppie di chiavi.

Non vengono usati public keyring o private keyring.

Il mittente e il destinatario devono condividere la passphrase attraverso un canale sicuro.

Il vantaggio è la semplicità.

Lo svantaggio è la distribuzione del segreto:
- se la passphrase viene intercettata, il messaggio può essere decifrato;
- se la passphrase è debole, può essere attaccata;
- se più persone devono ricevere il file, bisogna distribuire la passphrase a tutte.

Questo è il motivo per cui PGP/GPG usa normalmente un approccio ibrido:
- il contenuto viene cifrato simmetricamente;
- la chiave simmetrica viene cifrata con la chiave pubblica del destinatario.

### **13. Esercizio 2 — Traccia**

Creare due nuovi utenti nel sistema, **Alice** e **Bob**.

Per ciascun utente generare una coppia di chiavi da **1024 bit**.

Inoltre:
- fare importare ad Alice la chiave di Bob;
- fare importare a Bob la chiave di Alice;
- verificare la fingerprint;
- firmare la chiave.

In seguito:
- far inviare da Alice a Bob un messaggio cifrato con crittografia asimmetrica;
- far inviare da Bob ad Alice un messaggio sia cifrato che firmato;
- decifrare il messaggio e verificare la firma.

> ⚠️ La traccia PDF dice "Bob deve decifrare il messaggio e verificare la firma" dopo aver detto che Bob invia ad Alice un messaggio cifrato e firmato. Operativamente, se Bob invia ad Alice, il soggetto che decifra e verifica è Alice. Nella soluzione viene quindi risolto il flusso coerente: Bob invia, Alice decifra e verifica.

### **15. Esercizio 2 — Generazione della Chiave di Alice**

Entriamo come Alice:

```bash
sudo -iu alice
```

Generiamo la coppia di chiavi:

```bash
gpg --full-generate-key
```

Durante la procedura interattiva scegliere:
- tipo di chiave: RSA e RSA, se proposto;
- lunghezza: `1024` bit, come richiesto dall'esercizio;
- scadenza: a scelta per il laboratorio, per esempio `1y` oppure nessuna scadenza;
- nome reale: `Alice`;
- email: `alice@example.local`;
- commento: opzionale;
- passphrase: una passphrase robusta.

> ⚠️ L'esercizio richiede 1024 bit perché legato a un contesto didattico storico. Nella pratica moderna si usano chiavi più lunghe, per esempio 3072 o 4096 bit, se supportate e appropriate.

Verifichiamo la chiave di Alice:

```bash
gpg --list-keys
gpg --list-secret-keys
```

Visualizziamo la fingerprint:

```bash
gpg --fingerprint alice@example.local
```

La fingerprint deve essere annotata, perché Bob dovrà confrontarla quando importerà la chiave pubblica di Alice.

Usciamo dall'utente Alice:

```bash
exit
```

### **17. Esercizio 2 — Esportazione delle Chiavi Pubbliche**

Alice deve poter importare la chiave pubblica di Bob.

Bob deve poter importare la chiave pubblica di Alice.

Per questo bisogna esportare le chiavi pubbliche.

Come Alice:

```bash
sudo -iu alice
gpg --armor --export alice@example.local > alice-pub.asc
exit
```

Come Bob:

```bash
sudo -iu bob
gpg --armor --export bob@example.local > bob-pub.asc
exit
```

L'opzione:

```bash
--armor
```

produce una rappresentazione ASCII della chiave pubblica.

Questo è coerente con la logica PGP/OpenPGP: i dati devono poter passare anche attraverso canali testuali, come la posta elettronica.

Copiamo le chiavi pubbliche in una directory temporanea accessibile:

```bash
sudo cp /home/alice/alice-pub.asc /tmp/alice-pub.asc
sudo cp /home/bob/bob-pub.asc /tmp/bob-pub.asc
sudo chmod 644 /tmp/alice-pub.asc /tmp/bob-pub.asc
```

> 📌 Viene esportata solo la chiave pubblica. La chiave privata non deve mai essere consegnata all'altro utente.

### **18. Esercizio 2 — Alice Importa la Chiave di Bob**

Entriamo come Alice:

```bash
sudo -iu alice
```

Importiamo la chiave pubblica di Bob:

```bash
gpg --import /tmp/bob-pub.asc
```

Controlliamo che la chiave sia nel public keyring di Alice:

```bash
gpg --list-keys bob@example.local
```

Visualizziamo la fingerprint della chiave di Bob importata:

```bash
gpg --fingerprint bob@example.local
```

Alice deve confrontare questa fingerprint con quella comunicata da Bob attraverso un canale affidabile.

Esempi di canale affidabile:
- confronto di persona;
- telefonata verificata;
- canale istituzionale già autenticato;
- documento firmato o controllato separatamente.

Se la fingerprint coincide, Alice ha buona evidenza che la chiave pubblica importata sia davvero quella di Bob.

> ⚠️ Importare una chiave pubblica non basta: senza controllo della fingerprint, Alice potrebbe cifrare per una chiave dell'attaccante.

---

### **19. Esercizio 2 — Alice Firma la Chiave di Bob**

Dopo aver verificato la fingerprint, Alice può firmare la chiave di Bob:

```bash
gpg --sign-key bob@example.local
```

GPG chiede conferma.

Potrebbe anche chiedere la passphrase della chiave privata di Alice, perché la firma della chiave di Bob viene prodotta usando la chiave privata di Alice.

Questa operazione corrisponde alla certificazione della chiave nel modello web of trust.

Alice sta affermando:

> ho verificato che questa chiave pubblica appartiene a Bob.

Per una firma solo locale, non destinata a essere esportata verso altri utenti, si può usare:

```bash
gpg --lsign-key bob@example.local
```

Controlliamo le firme:

```bash
gpg --check-sigs bob@example.local
```

Usciamo da Alice:

```bash
exit
```

> 📌 La firma della chiave è distinta dalla firma di un messaggio: qui Alice firma la chiave pubblica di Bob per certificarne l'associazione con l'identità.

### **21. Esercizio 2 — Alice Invia a Bob un Messaggio Cifrato Asimmetricamente**

Entriamo come Alice:

```bash
sudo -iu alice
```

Creiamo un messaggio:

```bash
cat > messaggio-per-bob.txt <<'EOF'
Ciao Bob,
questo messaggio è cifrato con la tua chiave pubblica.
Solo tu, usando la tua chiave privata, dovresti poterlo leggere.
Alice
EOF
```

Cifriamo il messaggio per Bob:

```bash
gpg --armor --encrypt \
    --recipient bob@example.local \
    --output messaggio-per-bob.asc \
    messaggio-per-bob.txt
```

Significato:
- `--encrypt` indica cifratura asimmetrica OpenPGP;
- `--recipient bob@example.local` seleziona la chiave pubblica di Bob;
- `--armor` produce output ASCII;
- `--output messaggio-per-bob.asc` indica il file cifrato.

Il file cifrato può essere copiato nella home di Bob:

```bash
exit
sudo cp /home/alice/messaggio-per-bob.asc /home/bob/
sudo chown bob:bob /home/bob/messaggio-per-bob.asc
```

> 📌 Alice cifra con la chiave pubblica di Bob. Non serve conoscere la chiave privata di Bob.

### **22. Esercizio 2 — Bob Decifra il Messaggio di Alice**

Entriamo come Bob:

```bash
sudo -iu bob
```

Bob decifra il messaggio:

```bash
gpg --output messaggio-da-alice.txt --decrypt messaggio-per-bob.asc
```

GPG chiede la passphrase della chiave privata di Bob.

Se la passphrase è corretta, Bob ottiene:

```text
messaggio-da-alice.txt
```

e può leggerlo:

```bash
cat messaggio-da-alice.txt
```

Interpretazione crittografica:
- il contenuto non è stato cifrato direttamente con RSA per intero;
- GPG ha generato una chiave simmetrica di sessione;
- ha cifrato il messaggio con la chiave di sessione;
- ha cifrato la chiave di sessione con la chiave pubblica di Bob;
- Bob usa la propria chiave privata per recuperare la chiave di sessione;
- poi usa la chiave di sessione per recuperare il testo.

> ✅ Questo realizza la confidenzialità: solo chi possiede la chiave privata corrispondente alla chiave pubblica di Bob può decifrare il messaggio.

Usciamo da Bob:

```bash
exit
```

### **23. Esercizio 2 — Bob Invia ad Alice un Messaggio Cifrato e Firmato**

Entriamo come Bob:

```bash
sudo -iu bob
```

Creiamo un messaggio per Alice:

```bash
cat > messaggio-per-alice.txt <<'EOF'
Ciao Alice,
questo messaggio è cifrato per te e firmato da me.
La cifratura protegge la confidenzialità.
La firma permette di verificare integrità e autenticità.
Bob
EOF
```

Bob cifra e firma il messaggio:

```bash
gpg --armor --encrypt --sign \
    --local-user bob@example.local \
    --recipient alice@example.local \
    --output messaggio-per-alice.asc \
    messaggio-per-alice.txt
```

Significato:
- `--encrypt` cifra il messaggio per Alice;
- `--sign` firma il messaggio con la chiave privata di Bob;
- `--local-user bob@example.local` indica quale chiave usare per firmare;
- `--recipient alice@example.local` indica la chiave pubblica del destinatario;
- `--armor` produce output ASCII.

GPG può chiedere la passphrase di Bob perché deve usare la chiave privata di Bob per firmare.

Poi copiamo il messaggio nella home di Alice:

```bash
exit
sudo cp /home/bob/messaggio-per-alice.asc /home/alice/
sudo chown alice:alice /home/alice/messaggio-per-alice.asc
```

> 📌 Il messaggio è cifrato con la chiave pubblica di Alice e firmato con la chiave privata di Bob.

### **24. Esercizio 2 — Alice Decifra e Verifica la Firma**

Entriamo come Alice:

```bash
sudo -iu alice
```

Alice decifra il messaggio:

```bash
gpg --output messaggio-da-bob.txt --decrypt messaggio-per-alice.asc
```

Durante questa operazione GPG:
- usa la chiave privata di Alice per recuperare la chiave di sessione;
- decifra il messaggio;
- usa la chiave pubblica di Bob per verificare la firma;
- mostra a terminale l'esito della verifica.

Se tutto è corretto, GPG stampa un messaggio simile a:

```text
Good signature from "Bob <bob@example.local>"
```

Poi Alice può leggere il messaggio:

```bash
cat messaggio-da-bob.txt
```

Se GPG segnala una firma valida ma non completamente fidata, significa che la firma crittografica è corretta, ma il modello di fiducia locale non considera ancora completamente affidabile quella chiave.

Per questo nell'esercizio è importante:
- importare la chiave pubblica;
- controllare la fingerprint;
- firmare la chiave.

> ✅ In un solo comando, Alice ottiene confidenzialità tramite decifratura e autenticità/integrità tramite verifica della firma.

### **25. Spiegazione del Flusso Firma + Cifratura**

Quando Bob firma e cifra per Alice, GPG applica il modello studiato nella lezione PGP.

Dal lato di Bob:
- calcola il digest del messaggio;
- firma il digest con la chiave privata di Bob;
- aggiunge la firma al messaggio;
- comprime il contenuto;
- genera una chiave simmetrica di sessione;
- cifra il contenuto con la chiave di sessione;
- cifra la chiave di sessione con la chiave pubblica di Alice;
- produce un messaggio OpenPGP ASCII-armored.

Dal lato di Alice:
- individua la propria chiave privata tramite il key identifier;
- usa la passphrase per sbloccare la chiave privata;
- decifra la chiave di sessione;
- decifra il contenuto;
- decomprime;
- recupera la firma;
- recupera la chiave pubblica di Bob dal public keyring;
- verifica il digest firmato;
- confronta il digest con quello calcolato sul messaggio ricevuto.

Se la verifica riesce, Alice sa che:
- il messaggio non è stato modificato dopo la firma;
- il messaggio è stato firmato dalla chiave privata corrispondente alla chiave pubblica di Bob;
- solo Alice poteva decifrare il contenuto, perché cifrato per la sua chiave pubblica.

> ⚠️ La verifica "Bob ha firmato" è forte solo se Alice ha verificato che la chiave pubblica usata appartenga davvero a Bob.

### **26. Errori Comuni negli Esercizi**

#### **26.1. Confondere Chiave Pubblica e Privata**

Per cifrare verso Bob si usa la chiave pubblica di Bob.

Per firmare come Alice si usa la chiave privata di Alice.

Non bisogna mai inviare o copiare la chiave privata all'altro utente.

#### **26.2. Importare la Chiave senza Verificare la Fingerprint**

Importare una chiave da file non prova che appartenga davvero al soggetto dichiarato.

La fingerprint deve essere confrontata tramite un canale affidabile.

#### **26.3. Dimenticare di Firmare la Chiave Importata**

La firma della chiave importata serve a costruire fiducia locale.

Senza firma, GPG può cifrare comunque, ma potrebbe mostrare avvisi di fiducia.

#### **26.4. Usare l'Utente Linux Sbagliato**

Se si esegue un comando come `root`, si usa il keyring di `root`, non quello di Alice o Bob.

Per questo è importante entrare correttamente con:

```bash
sudo -iu alice
```

oppure:

```bash
sudo -iu bob
```

#### **26.5. Scambiare Mittente e Destinatario**

Se Bob invia ad Alice, il messaggio va cifrato con la chiave pubblica di Alice e firmato con la chiave privata di Bob.

Alice decifra con la propria chiave privata e verifica con la chiave pubblica di Bob.

### **27. Riepilogo**

Gli esercizi mostrano due usi fondamentali di GPG.

Nel primo esercizio si usa la cifratura simmetrica:
- un file viene cifrato con AES, 3DES e Blowfish;
- la stessa passphrase viene usata per decifrare;
- non intervengono chiavi pubbliche o private.

Nel secondo esercizio si usa OpenPGP con coppie di chiavi:
- Alice e Bob generano una coppia di chiavi da 1024 bit;
- esportano le rispettive chiavi pubbliche;
- importano la chiave pubblica dell'altro;
- verificano le fingerprint;
- firmano le chiavi importate;
- Alice invia a Bob un messaggio cifrato;
- Bob invia ad Alice un messaggio cifrato e firmato;
- Alice decifra e verifica la firma.

Il collegamento con la teoria PGP è diretto:
- la cifratura asimmetrica protegge la chiave di sessione;
- la cifratura simmetrica protegge il contenuto;
- la firma digitale protegge integrità e autenticità;
- la fingerprint aiuta ad autenticare una chiave pubblica;
- il keyring memorizza chiavi e fiducia;
- la web of trust sostituisce una gerarchia centralizzata di Certification Authority.

> ✅ GPG rende operativi i concetti di PGP: protezione simmetrica dei dati, uso di chiavi pubbliche e private, firma digitale, verifica della fingerprint e gestione decentralizzata della fiducia.
